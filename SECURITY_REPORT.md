**Date:** 2026-04-11

**Assessed by:** Manual code review

---

## 1. Executive Summary

The overall security posture has **improved significantly** since the last review. All seven previously-prioritised findings (F-01 through F-07) have been remediated — including the on-disk credential file, OAuth Testing-mode expiry, GitHub Actions SHA pinning, the unused `id-token: write` permission, the workflow shell interpolation risk, missing job timeout, and dependency hash verification.

The two new features added since the last report — the **Card Maker** (browser-based story generation) and the **Cloudflare Worker proxy** — are implemented with solid security thinking: the API key stays server-side, CORS is restricted to known origins, rate limiting is server-side (cannot be bypassed by the browser), and all Claude-returned content is escaped or sanitised before being inserted into the DOM.

Four items remain open, all **Low or Informational**. There are no Critical or High findings. One new Low finding was identified (Worker URL is public, enabling direct API calls that bypass CORS).

## 3. Remediated Findings (since 2026-04-08)

### F-01 — `client_secrets.json` deleted ✓
The file no longer exists on disk. Confirmed via filesystem check.

### F-02 — OAuth consent screen published ✓
The Google Cloud OAuth consent screen has been moved to Production mode. Refresh tokens are now long-lived — no more 7-day expiry.

### F-03 — All GitHub Actions SHA-pinned ✓
All three actions in `daily.yml` are now pinned to full commit SHAs:
- `actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683` (v4.2.2)
- `actions/setup-python@0b93645e9fea7318ecaed2b359559ac225c90a2b` (v5.3.0)

### F-04 — `id-token: write` removed ✓
The workflow permissions block now only contains `contents: write` and `pages: write` — the minimum required.

### F-05 — Workflow input uses environment variable ✓
`github.event.inputs.rebuild_only` is now routed through `env: REBUILD_ONLY:` before the shell `if` statement, eliminating the interpolation risk.

### F-06 — Job timeout added ✓
`timeout-minutes: 15` is present on the `generate` job.

### F-07 — Dependency hash verification active ✓
`requirements.txt` is generated by `pip-compile --generate-hashes`, which annotates every package with `--hash=sha256:...` lines. When all packages in a requirements file carry hash annotations, `pip install -r requirements.txt` automatically enables hash-checking mode — the download is verified against the known fingerprint before installation. No `--require-hashes` flag is needed separately.

## 4. Open Findings

### F-08 / LOW — No branch protection on gh-pages

**Description:** The `gh-pages` branch, which serves the public-facing HTML via GitHub Pages, has no branch protection rules.

**Risk:** For a single-developer personal repo, the practical risk is minimal. If a collaborator were ever added, they could push arbitrary HTML to the public site without review.

**Recommendation:** In GitHub repo settings, add a branch protection rule for `gh-pages` that restricts pushes to the Actions bot.

---

### F-09 / LOW — No secret rotation schedule

**Description:** No documented or automated schedule exists for rotating the Anthropic API key, Gmail OAuth credentials, or the Worker's copy of the Anthropic key.

**Risk:** Long-lived credentials widen the exposure window if leaked.

**Recommendation:** Set a calendar reminder for annual rotation. Document the rotation steps (update GitHub secret + `wrangler secret put ANTHROPIC_API_KEY`) so it can be done quickly if a leak is suspected.

---

### F-10 / INFORMATIONAL — SVG sanitisation via regex deny-list

**Description:** Both `sanitize_svg()` (Python, server-side) and `cmSanitizeSvg()` (JS, Card Maker) use regex-based deny-lists to strip dangerous SVG constructs. The Python version also strips `<foreignObject>` and `<image>` elements; the JS version does not (see F-14).

**Risk:** Regex-based HTML/SVG sanitisation is inherently fragile — novel encoding tricks could evade the patterns. That said: the SVG source is Claude's own output (not untrusted user input), the CSP blocks image loads (`img-src`) and restricts outbound connections, and the audience is a single trusted person.

**Recommendation:** Acceptable for the current threat model. If the project ever accepts untrusted SVG input, switch to an allowlist-based parser.

### F-11 / INFORMATIONAL — Hardcoded recipient email

### F-13 / LOW — Worker URL is public; non-browser callers bypass CORS

**Description:** The Cloudflare Worker URL (`https://daily-digest-proxy.dizchrisctrl.workers.dev`) is baked into the generated HTML and is visible in the page source of the public GitHub Pages site. CORS restrictions in the Worker only apply to browser requests — any `curl` or scripted HTTP client can POST directly to the Worker and consume the Anthropic API key.

**Risk:** An adversary who discovers the URL could make up to 10 Claude API calls per week (the rate limit) without going through the browser. At Claude Opus pricing, the maximum weekly cost exposure from abuse is approximately $2–8 depending on token usage. The rate limit directly constrains the blast radius.

**Recommendation:** This is an accepted trade-off of the design — adding authentication would require user accounts. The rate limit is the appropriate mitigation here. If abuse were observed, the Worker URL can be changed (rotate the `WORKER_URL` GitHub secret and redeploy the Worker) to invalidate any cached URL.

**Note:** CORS does provide meaningful protection against one class of attack — it prevents other websites from making API calls via the *visitor's* browser when they visit a malicious site. The CORS restrictions correctly address that threat.

### F-14 / INFORMATIONAL — `cmSanitizeSvg` less thorough than Python counterpart

**Description:** The Python `sanitize_svg()` strips `<script>`, `<foreignObject>`, `<image>`, `on*` handlers, and `javascript:` hrefs. The JS `cmSanitizeSvg()` (used in Card Maker) strips `<script>`, `on*` handlers, and `javascript:` hrefs — but does **not** strip `<foreignObject>` or `<image>` elements.

**Risk:** Low. A crafted SVG from Claude could embed a `<foreignObject>` containing HTML, or an `<image>` pointing to an external resource. However:
- `img-src 'none'` in CSP blocks all image loads.
- `foreignObject` rendering is blocked by `default-src 'none'` and no `frame-src`.
- The SVG source is Claude's tool_use output, not raw user input.

**Recommendation:** Add `<foreignObject>` and `<image>` element removal to `cmSanitizeSvg` to match the Python version:
```js
function cmSanitizeSvg(s) {
  return s
    .replace(/<script[\s\S]*?<\/script>/gi, '')
    .replace(/<foreignObject[\s\S]*?<\/foreignObject>/gi, '')
    .replace(/<image[^>]*>/gi, '')
    .replace(/\s+on\w+\s*=\s*["'][^"']*["']/gi, '')
    .replace(/href\s*=\s*["']javascript:[^"']*["']/gi, '')
    .replace(/xlink:href\s*=\s*["']javascript:[^"']*["']/gi, '');
}
```

### F-15 / INFORMATIONAL — Article text passes to Claude without sanitisation

**Description:** In the Card Maker, article text (either fetched via the Worker or pasted by the user) is passed directly into Claude's prompt without stripping potential prompt-injection attempts (e.g., `\n\nIgnore all previous instructions...`).

**Risk:** Minimal. Claude's `tool_use` with `tool_choice: {type: 'tool', name: 'publish_story'}` forces a structured JSON response — Claude cannot output free-form instructions or exfiltrate data to arbitrary locations. Any prompt injection would at worst produce a garbled or unexpected story card, not a security breach. The only destination for Claude's output is the rendered card in the DOM (which is fully escaped).

**Recommendation:** No action required for the current threat model. If the card content were ever stored, shared, or acted upon programmatically, adding newline stripping to the article text would be a sensible defence-in-depth step.

## 5. What's Solid

The application demonstrates strong security practices throughout:

- **All previous High/Medium findings resolved.** The rapid remediation of F-01 through F-06 after the last report reflects good security hygiene.
- **API key never in the browser.** The Cloudflare Worker proxy holds the Anthropic key server-side. The browser page contains only the Worker URL — a public endpoint with rate limiting.
- **Rate limiting is server-side.** Cloudflare KV is authoritative — cannot be manipulated from the browser.
- **All Claude output is escaped.** Both the daily build path (`esc()` in Python) and the Card Maker path (`h()` in JS) consistently escape all dynamic values before DOM insertion.
- **SVG content is sanitised** on both paths before inline embedding.
- **CSP is tightly scoped.** `default-src 'none'` with `connect-src` restricted to the single Worker URL. Every other resource type is blocked.
- **CORS is origin-restricted** in the Worker to the exact production domain + localhost.
- **Dependencies are hash-verified** via pip-compile with `--generate-hashes`.
- **All Actions SHA-pinned.**
- **Secrets are stored in GitHub Actions Secrets** — not in code, not in config files.
- **Gmail scope is `gmail.send` only** — no inbox access possible.
- **External links use `rel="noopener noreferrer"`.**
- **Feed fetching has a 15-second timeout.**
- **Prior credential leak was handled correctly** — old OAuth client deleted, new credentials generated.
