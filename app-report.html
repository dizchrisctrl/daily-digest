App Documentation

Daily Digest

A fully automated pipeline that fetches tech and security news, generates AI-written deep-dive stories, and delivers them as a GitHub Pages site and email — every morning at 7 AM.

Language: Python 3.12
AI Model: Claude Opus 4.6
Schedule: Daily 7:00 AM EST
Hosting: GitHub Pages
Email: Gmail API (send-only)
Proxy: Cloudflare Worker
Rate Limit: 10 cards/week (KV)
Repo: dizchrisctrl/daily-digest

Pipeline Steps
1
Fetch Articles
fetch_articles()
Polls 31 RSS feeds across three categories (10 AI, 11 Cyber, 10 Notables). Only articles published in the last 48 hours are considered — older entries are skipped. Caps at 2 entries per feed and a 15-second socket timeout. Strips HTML from summaries and sanitises titles to single lines to prevent prompt injection.
Think of this as a news clipping service with a freshness filter. It visits 31 trusted news sites, grabs only the latest headlines (within 48 hours), and tidies them up before passing them along. The 15-second timeout means it won't get stuck waiting forever if a site is down.
2
Select Top Articles
call_claude_for_section() — selection pass
Sends up to 8 article titles to Claude Opus with a lightweight prompt asking it to return 3 indices (the most notable stories). Uses a 50-token budget — no story content yet.
Before spending time writing full stories, the app asks Claude to quickly scan the headlines and pick the three most interesting ones — like asking an editor to skim the wire and flag the top stories before assigning writers.
3
Generate Stories
call_claude_for_section() × 3
One Claude Opus API call per story (3 per section, 7 total including Notables). Each call uses tool_use with a strict JSON schema — forcing structured output for every field including the SVG diagram. Max 8,000 tokens per call.
Instead of asking Claude to write a free-form article, the app hands it a form to fill in. Claude must supply a headline, summary, diagram, quiz questions, and more — all in a specific format. This makes the output reliable and machine-readable, not just a wall of text. One story per API call avoids hitting Claude's output size limit.
4
Generate Notables
call_claude_for_notables()
Single call returning 5 shorter highlights (headline, summary, applicability, category) from a broader pool drawn from 10 feeds including The Verge, Wired, Reuters, Hacker News, IEEE Spectrum, Fast Company, WSJ Tech, NYT Tech, TechCrunch, and MIT Tech Review.
The Notables section covers broader tech and world news — not just AI or security. It's a lighter format: a short summary and a note on why it matters to someone in tech. Claude picks the 5 most relevant from the pool and returns them all in one go.
5
Build HTML
generate_html() + save_output()
Assembles the full interactive webpage from a Python string template. Injects escaped story data, inline SVG diagrams, the Cloudflare Worker URL (baked from WORKER_URL env var), and a static JS block for tabs, keyboard navigation, deep-link sharing with highlight animation, TTS, and the Card Maker. Writes output/index.html, output/digest.json, output/guide.html, and per-story OG redirect pages under output/s/.
Once all stories are ready, the app assembles them into a web page — the same way you'd slot pieces into a template document. Everything is baked into a single HTML file with no external requests needed, so it works reliably as a static site.
6
Send Email
send_email()
Composes a short HTML digest email (headlines + TLDRs + a link to the full page) and sends it via the Gmail API using OAuth2 refresh-token flow. No password — send-only scope, cannot read the inbox.
The app sends email the same way a trusted app like Notion or Todoist would — it has a permission token that only allows sending, not reading. No password is ever stored. The token is kept in GitHub's encrypted secrets vault and never touches the code.
7
Deploy to Pages
peaceiris/actions-gh-pages
The output/ folder is pushed to the gh-pages branch by a SHA-pinned GitHub Action, replacing the previous day's content. The site is publicly accessible at the configured Pages URL.
GitHub Pages works like a simple file server — whatever is in a special branch gets served as a website. This step takes today's freshly built HTML file and publishes it there, overwriting yesterday's version. The site always shows the latest digest.
Card Maker (Interactive)
✨
On-Demand Story Cards
Card Maker tab · browser-based
The Card Maker tab lets any visitor paste an article URL and generate a full story card using the same format as the daily digest — headline, TL;DR, SVG diagram, Insights, Devil's Advocate, Deep Dive, and more. All processing happens through the Cloudflare Worker proxy.
Instead of waiting for the daily run, you can generate a story card for any article you find interesting — right now, in the browser. Paste the URL, click Generate, and the same AI that writes the morning digest creates a card in the same format.
☁
Cloudflare Worker Proxy
daily-digest-proxy.dizchrisctrl.workers.dev
A serverless edge function deployed to Cloudflare. Handles two tasks: (1) fetches article HTML server-side to bypass CORS restrictions, (2) proxies Anthropic API calls so the API key is never exposed in the browser. The Worker URL is baked into the page at build time from the WORKER_URL GitHub secret.
A Cloudflare Worker is like a tiny server that runs at the edge of the internet — extremely fast and free for low traffic. The Card Maker never calls Anthropic or the article URL directly from your browser. Everything goes through the Worker, which holds the API key securely and adds proper headers.
⏳
Weekly Rate Limiting
Cloudflare KV · 10 cards / week
The Worker tracks card generation count in Cloudflare KV (a key-value store). Key = Monday date of the current week (e.g. cards_2026_04_13). Counter increments on each successful API call. Returns HTTP 429 with a friendly message when the weekly limit of 10 is reached. Resets automatically every Monday. Keys expire after 14 days for self-cleanup.
KV is Cloudflare's simple key-value database. The Worker reads and increments a counter stored there on each card request. Because this check happens on the server, it can't be bypassed by the browser. The key auto-deletes after 2 weeks so the database stays clean.

AI Story Fields

Each deep-dive story (AI section and Cybersecurity section) is a structured object. Claude Opus must populate every required field via the publish_story tool.

FieldTypeDescription
headlinestringShort punchy title — never empty
tldrstringOne sentence that tells the whole story
why_it_mattersstring2–3 sentences on real-world significance
concept_titlestringThe core technical concept being illustrated
concept_explainedstring4 paragraphs: analogy → how it works → tie to story → broader implications
visual_svgstringComplete <svg> diagram (700×340). Types: attack chain, architecture, data flow, timeline, comparison. Dark theme, accent colour per section.
public_opinionstringWhat HN, Reddit, security Twitter are saying about this — collapsible section in the page
opinion_assessmentstringCritical analysis of public opinion — what's valid, overblown, or missing
devils_advocatestringA sharp counter-perspective that challenges the dominant sentiment — an overlooked irony or reframe that makes the reader stop and think differently
quizarray[3]Three Insight cards, each with a lens (Scientific / Historical / Societal), a thought-provoking hook (q), a crisp key insight (a), and a 2–3 sentence explanation (explain)
deep_divestringA Socratic question connecting the story to bigger trends
deep_dive_impactstring2–3 sentences on how this story directly affects the reader — their work, tools, security posture, or career. Personal and specific.
deep_dive_outlookstring2–3 sentences on what is likely to happen next — what trends this story accelerates or disrupts, what to watch in the coming weeks or months
source_url / sourcestringOriginal article URL and publication name
tech_tagsarray[0–3]Clickable tag objects with name, description, and relevance. Only specific tech — no generic terms like "AI" or "cloud".
affected_systemsarrayFor CVE/vulnerability stories: system name + affected version range. Empty for non-vulnerability stories.
Output Files
📄
index.html
output/index.html → gh-pages branch
The full interactive digest page. Single self-contained file — no CDN dependencies, no external requests. Includes tabbed navigation (AI / Cyber / Notables), keyboard shortcuts (j/k, 1/2/3, Enter, Escape), tag modals, scroll progress bar, and inline SVG diagrams.
The main deliverable — a complete web page baked into one file. Open it in any browser with no internet connection and everything still works. All styles, scripts, and images are embedded directly. This makes it fast, private, and reliable.
{}
digest.json
output/digest.json → gh-pages branch
The raw structured data for the day's digest. Preserved on the gh-pages branch so the --rebuild mode can regenerate the HTML without calling Claude again — useful for template changes and debugging.
The "source of truth" data file — all stories in structured JSON. If you update the page design, you can rebuild the HTML from this file instantly without paying for another Claude API call. It's also what you'd use if you wanted to build a different frontend.
📖
guide.html
output/guide.html → gh-pages branch
The reader guide explaining how the digest works, how it's secured, and how to use Card Maker. Built from daily-rundown-explainer.html — copied as-is into the output folder so it's accessible at the site URL alongside the main digest.
A companion page that lives at the same site URL. Readers can open it to understand what they're looking at, how the AI works, and how sharing and Card Maker work. It's a static copy so no extra build step is needed.
🔗
/s/story-*.html
output/s/ → gh-pages branch
Per-story OG redirect pages — one per story (e.g. s/story-ai-1.html). Each has Open Graph meta tags (title, description, image) for rich link previews when shared on social media, plus a <meta refresh> and JS redirect that sends visitors to the correct story on the main digest page, auto-expanding and highlighting the card.
When you share a story link, social media platforms read the OG tags from this redirect page to generate a rich preview card (headline + description). When a user clicks the link, they land here for a fraction of a second, then get sent straight to the correct story on the main page — which opens and highlights automatically.
Email Delivery
✉
Gmail API
OAuth2 · gmail.send scope only
Sends an HTML summary email containing today's headlines, TLDRs, and a link to the full digest. Authenticates using a refresh token — no password stored anywhere. The OAuth consent screen is published (Production mode), so the refresh token does not expire.
Rather than storing a Gmail password, the app uses a permission token that Google issued after you clicked "Allow" during setup. This token can only send emails — it cannot read, search, or delete anything. Google can revoke it at any time from your account settings, and the app would immediately stop working until re-authorized.
🔐
OAuth2 Credentials
Stored in GitHub Actions Secrets
Four secrets: GMAIL_ADDRESS, GMAIL_CLIENT_ID, GMAIL_CLIENT_SECRET, GMAIL_REFRESH_TOKEN. Injected as environment variables at runtime — never written to disk, never logged.
GitHub's Secrets are an encrypted vault — you put values in, and only your Actions workflows can read them back. The values are masked in all logs. It's the equivalent of keeping your keys in a safe rather than taped to the front door.
Infrastructure
⚫
GitHub Actions Workflow
.github/workflows/daily.yml
Runs on ubuntu-latest at 12:00 UTC (07:00 EST) daily. Steps: checkout → setup Python 3.12 → install deps → checkout gh-pages data → generate digest → deploy. Supports manual trigger with optional --rebuild flag.
GitHub Actions is a free cloud computer that runs your script on a schedule. Every morning at 7 AM, GitHub spins up a fresh Linux machine, runs the digest script, and throws the machine away when it's done. No server to maintain, no always-on cost.
🌐
GitHub Pages
gh-pages branch · public site
The gh-pages branch serves as the web root. Protected from deletions and force-pushes. Only the Actions bot can commit to it. Each run replaces the previous day's content.
GitHub Pages turns a Git branch into a website. Every time the workflow runs, it pushes the new HTML into that branch, which GitHub automatically serves at your Pages URL. It's free static hosting with zero config.
🔐
Actions Permissions
Minimal — contents:write, pages:write
The workflow only has the two permissions it actually needs. id-token: write (OIDC cloud auth) was explicitly removed. All three actions are pinned to full commit SHAs — not floating version tags — to prevent supply chain attacks.
Pinning an action to a commit SHA is like saying "I trust this exact version of the code, not whatever gets tagged v4 in the future." If someone hijacked the action repo and pushed malicious code under the same tag, your workflow would be unaffected because it's locked to a specific snapshot.
☁
Cloudflare Worker
cloudflare-worker/index.js · always-on edge
Serverless proxy deployed to Cloudflare's global edge network. CORS is restricted to dizchrisctrl.github.io, localhost, and 127.0.0.1. Two operational modes: fetch mode ({ fetch_url }) fetches article HTML server-side; proxy mode forwards signed Anthropic API requests. Rate limit enforced via Cloudflare KV binding RATE_LIMIT.
The Worker is a middleman that runs permanently at Cloudflare's edge — no server to maintain, no cost at low traffic. When the Card Maker needs to fetch an article or call Claude, it asks the Worker instead of going directly. The Worker adds the API key, enforces rate limits, and returns the result.
⏱
Job Timeout
timeout-minutes: 15
The job is hard-capped at 15 minutes. Normal runs complete in ~6 minutes. The timeout prevents a stuck API call or feed from burning Actions minutes against the monthly free quota.
Without a timeout, if the script ever got stuck in an infinite loop or a server stopped responding, GitHub would let it run for up to 6 hours — burning through your free compute minutes. The 15-minute cap acts as an automatic kill switch.

Security Assessment
✓
Overall Posture: Good — Pass with Notes
No critical or high findings active. Strong controls in place for a personal automated pipeline. Remaining items are low-severity hardening steps.
Output Escaping (XSS)
Secured
All dynamic values rendered into HTML pass through esc(). URLs are validated through safe_url() which enforces http:// or https:// scheme only, blocking javascript: URIs.
If an RSS feed injects something like <script>alert(1)</script> into a headline, esc() converts the angle brackets into harmless display characters before they reach the page. The browser sees text, not code.
Implemented in generate_html() and send_email()
SVG Sanitisation
Secured
sanitize_svg() strips <script>, <foreignObject>, <image>, inline on* event handlers, and javascript: hrefs from every Claude-generated SVG before it is embedded inline.
SVG is a vector image format but it can also run JavaScript. Even though Claude generates the SVG, this step scrubs out anything that could execute code — acting as a safety net regardless of what the AI produces.
Applied to all visual_svg fields before HTML insertion
Content Security Policy
Secured
A CSP meta tag on every generated page uses default-src 'none'. The only permitted outbound connection is connect-src [WORKER_URL] — the single Cloudflare Worker endpoint baked in at build time. All other external resources, frames, objects, and form submissions are blocked. Provides defence-in-depth against sanitisation bypasses.
Even if a malicious script somehow got past the sanitiser, the browser's CSP would block it from loading external resources or phoning home. The only thing the page can connect to is the Worker — and only the Worker. It's a second lock on the door.
default-src 'none' · connect-src = Worker URL only (baked at build time)
Card Maker Rate Limiting
Secured
Card generation is capped at 10 per week via a server-side counter in Cloudflare KV. The check happens in the Worker before any API call — it cannot be bypassed from the browser. Returns HTTP 429 with a reset date when the limit is reached. Keys expire after 14 days for automatic cleanup.
The rate limit lives on Cloudflare's servers, not in the browser — so there's no way for a visitor to modify the counter. If someone tried to call the Worker directly (bypassing the page), the limit would still apply because it's enforced before the API key is ever used.
10 cards/week · KV binding RATE_LIMIT · resets Monday
Prompt Injection Mitigation
Secured
Feed titles and source names are sanitised with _single_line() which strips newlines, tabs, and carriage returns before they reach Claude's prompt. Feed links are validated to http/https only.
A malicious RSS feed could try to inject fake instructions into the prompt — like embedding "\n\nIgnore all previous instructions and..." in an article title. Stripping newlines prevents this escape technique from working.
_single_line() applied to title and source fields
Gmail OAuth2 Scope
Secured
The app requests only the gmail.send scope — it cannot read, search, modify, or delete email. The consent screen is in Production mode, so refresh tokens do not expire after 7 days.
Scope is a way of limiting what a permission token can do. Even if someone stole the token, the worst they could do is send email as you — they couldn't read your inbox, access your calendar, or impersonate you to Google services.
Production mode — long-lived refresh tokens
Secrets Management
Secured
All six GitHub secrets (Anthropic API key, four Gmail OAuth values, Worker URL) are stored in GitHub Actions Secrets. Injected as environment variables — never written to disk, never printed in logs. The Cloudflare Worker holds a separate copy of ANTHROPIC_API_KEY set via wrangler secret put. client_secrets.json deleted from local disk after setup.
GitHub Secrets are encrypted at rest and masked in any log output — if a secret value ever appears in a log line, GitHub replaces it with ***. The client_secrets.json file that was used during initial setup has been deleted since the values are now safely in the vault.
client_secrets.json deleted · .gitignore excludes credential files
Actions SHA Pinning
Secured
All three GitHub Actions are pinned to full commit SHAs — not floating version tags. Prevents supply-chain attacks where a compromised action repo pushes malicious code under an existing tag (e.g. the tj-actions/changed-files incident, 2025).
A version tag like @v4 can be moved by the action's author at any time. A commit SHA like @11bd71901b... is immutable — it points to exactly one snapshot of code that can never change. Pinning to SHAs is the equivalent of checking a package's checksum before installing it.
checkout, setup-python, actions-gh-pages all SHA-pinned
Feed Timeout
Secured
A 15-second socket.setdefaulttimeout() is applied before any feed is fetched and restored afterward. Prevents a slow or unresponsive feed from hanging the entire workflow.
Without a timeout, if one news site stops responding, the script waits indefinitely. Setting a 15-second limit means the worst case is a 15-second delay per broken feed — the rest of the pipeline still runs normally.
socket.setdefaulttimeout(15) in fetch_articles()
External Link Safety
Secured
All external links in the generated HTML include rel="noopener noreferrer". This prevents tab-napping (the linked page cannot access window.opener) and stops the referrer header from leaking the digest URL.
When you open a link in a new tab without these attributes, the new page can technically control the tab you came from via JavaScript. noopener severs that connection. noreferrer also hides where you came from, so the news site doesn't see your digest URL in its analytics.
Applied to all target="_blank" links in template
Dependency Pinning
Low
All five packages are pinned to exact versions in requirements.txt (good). However, pip install does not use --require-hashes, meaning artifact integrity is not verified — only version identity.
Pinning the version number stops unexpected upgrades but doesn't verify the download is exactly what you expect. Hash verification would confirm the bytes you downloaded match a known fingerprint — catching a tampered package even if the version number is correct. Low risk for a personal project.
Versions pinned · hash verification not yet enabled
Secret Rotation
Low
No documented rotation schedule for the Anthropic API key or Gmail OAuth credentials. Long-lived credentials widen the exposure window if they are ever leaked.
Rotating credentials means generating new keys and revoking old ones on a schedule — even if there is no known leak. It limits how long any stolen credential remains useful. Annual rotation is a reasonable baseline for a low-risk personal project.
Recommended: annual rotation reminder
Previous Credential Leak
Remediated
An earlier set of OAuth credentials was inadvertently shared in a chat session. The compromised OAuth client was deleted in Google Cloud Console and entirely new credentials were generated and stored.
The correct response to a credential leak is to revoke first, rotate second. Deleting the entire OAuth client (not just the secret) ensures the old credentials are permanently useless — there is no race condition where an attacker can use them between revocation and rotation.
Old client deleted · new credentials generated and stored
