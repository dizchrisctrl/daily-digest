🤖 What Claude Does

For each article, Claude writes a headline, TL;DR, full concept explanation, SVG diagram, Devil's Advocate counter-take, three Insights (scientific, historical, societal lenses), public opinion breakdown, and a personal impact + outlook section.

📄 What Gets Built

index.html — the interactive website with tabs, expandable cards, dark/light mode, and keyboard navigation.

digest.json — the raw data, used to rebuild HTML without re-calling Claude.

guide.html — the reader guide (this page).

/s/story-*.html — per-story redirect pages for shareable deep-links.

📬 The Email

After the page is built, a summary email is sent to your Gmail via the Gmail API. It contains headlines, TL;DRs, and a link to the full digest.

When does it run? Automatically every day at noon UTC (8 AM EDT / 7 AM EST). You can also trigger it manually from GitHub at any time.
02   Card Maker

Generate a Story Card for Any Article

The Card Maker is a fourth tab on the digest page that lets you turn any article on the internet into a full story card — same format as the daily digest — on the spot. No account, no API key required.

✨ What you get

The exact same card format as the daily stories: headline, TL;DR, concept explainer, SVG diagram, public opinion breakdown, Devil's Advocate counter-take, three Insight cards, and a Deep Dive with impact and outlook.

💰 Does it cost anything?

Nothing. The AI calls go through a private proxy. You're limited to 10 cards per week (resets every Monday) to keep costs in check — but within that limit it's completely free to use.

How to Use It

  1. 1Click the Card Maker tab at the top of the digest page (the pink one).
  2. 2Paste an article URL into the URL field — any news article, blog post, or public page works. Or skip the URL and paste the article text directly into the text box below it.
  3. 3Click Generate Card and wait about 20 seconds. Claude reads the article and writes the full card.
  4. 4The card appears on the page with all the same sections as a daily story — scroll down to read it, expand sections, or listen with the TTS button.
  5. 5Use Copy HTML to grab the card's source code, or ✕ Clear to start fresh with a new article.
URL fetch failing? Some sites block automated fetches. If you see a "failed to fetch" error, copy and paste the article text directly into the text box instead — that path always works.

The Weekly Limit

⏳ 10 cards per week

Each card generation uses one call to Claude Opus, which costs a small amount. The limit caps usage at 10 cards per week to keep the costs predictable. After generating a card the page shows how many you have left.

The counter resets every Monday. If you hit the limit early you'll see a message telling you the exact reset date.

The limit is enforced server-side — it applies regardless of which browser or device you use.

Best Articles to Use It On

Works great

Tech news, security advisories, AI research announcements, product launches, policy changes — anything with a clear subject and enough substance for Claude to explain.

Works less well

Paywalled articles (the fetch will fail or return a login wall), opinion pieces with no factual core, very short items, or pages that are mostly video/image with little text.

03   Authentication

How Credentials Work

The app needs to talk to two external services — Claude AI (Anthropic) and Gmail. Both require secret keys that prove the app has permission. Here's how each one works.

🔑 Anthropic API Key

What it is: A long string of characters that gives the app permission to use Claude AI. Think of it like a password for the AI service.

Where it lives: Stored as a GitHub Secret named ANTHROPIC_API_KEY. Never written in the code.

How it's used: The workflow injects it as an environment variable when the script runs. The script reads it with os.environ["ANTHROPIC_API_KEY"].

📧 Gmail OAuth2 — How It Works

Gmail doesn't let apps use your password directly. Instead it uses OAuth2 — a system where you grant specific, limited permissions. Your app was granted send-only access, meaning it can send email but can never read, search, or touch your inbox.

# The four Gmail secrets stored in GitHub:
GMAIL_ADDRESS = your Gmail address
GMAIL_CLIENT_ID = identifies your Google Cloud app
GMAIL_CLIENT_SECRET = proves you own that app
GMAIL_REFRESH_TOKEN = long-lived token that gets new access tokens

How tokens work: OAuth2 gives out two types of tokens. An access token is short-lived (1 hour) and lets the app act. A refresh token is long-lived and gets new access tokens automatically. You stored the refresh token — so the app refreshes itself every time it runs without needing you to log in again.

☁️ Cloudflare Worker (Card Maker)

What it is: A small serverless function deployed to Cloudflare's edge network at daily-digest-proxy.dizchrisctrl.workers.dev. It acts as a secure proxy between the Card Maker in your browser and the Anthropic API — your API key never leaves the server.

Secrets involved: The Worker holds its own copy of ANTHROPIC_API_KEY (set via wrangler secret put). The GitHub repo holds a WORKER_URL secret (the Worker's URL) which gets baked into the generated page at build time.

Rate limiting: The Worker tracks how many cards have been generated this week using Cloudflare KV (a key-value store). Limit is 10 cards per week — resets every Monday. Returns a 429 error with a friendly message when the limit is reached.

Setup was a one-time thing. You ran setup_gmail_oauth.py locally once — it opened a browser, you clicked Allow, and it printed the four values to your terminal. You pasted those into GitHub Secrets. The Worker was deployed once with wrangler deploy from the cloudflare-worker/ folder. Neither needs to be done again unless you re-authorize or re-deploy.
04   GitHub — The Basics

What GitHub Is and How It Relates to Your Page

GitHub is a platform that does three things for your project: stores your code safely in the cloud, runs automated tasks, and hosts your website for free.

📁 Repository

A repository (repo) is your project's folder in the cloud. Everything — code, workflow, history — lives at github.com/dizchrisctrl/daily-digest.

💾 Commit

A commit is a saved snapshot of your code at a point in time. Every time you git commit, you create a permanent record of what changed and why.

🌿 Branch

A branch is a separate version of your code. You have two: master (your source code) and gh-pages (the built website files).

⬆️ Push / ⬇️ Pull

Push sends your local commits up to GitHub. Pull brings changes from GitHub down to your machine. These keep the two in sync.

⚙️ GitHub Actions

Automated workflows defined in .github/workflows/daily.yml. They run on a schedule or on demand — this is what generates the digest every morning.

🌐 GitHub Pages

Free website hosting built into GitHub. It serves files from your gh-pages branch as a public website at dizchrisctrl.github.io/daily-digest.

Your Two Branches Explained

🌿 master — your source code

This is where all the actual code lives: generate_digest.py, requirements.txt, and the workflow file. When you edit code and push, it goes here. The workflow reads from this branch when it runs.

🌿 gh-pages — your live website

This branch only contains the built output: index.html and digest.json. You never edit this directly — the workflow pushes here automatically after each run. GitHub Pages serves this branch as your public website.

How the Workflow Runs

# What happens every morning automatically:

1. GitHub Actions checks out your master branch
2. Installs Python dependencies from requirements.txt
3. Injects secrets as environment variables (including WORKER_URL)
4. Runs generate_digest.py → fetches news → calls Claude → builds HTML
   The WORKER_URL is baked into the page so Card Maker knows where to call
5. Pushes index.html, digest.json, guide.html, and /s/ redirect pages to gh-pages
6. GitHub Pages detects the new push and rebuilds your live site
7. Email is sent to your inbox

06   Security Assessment

How Your Page Is Secured

Here's a plain-English breakdown of the security controls in place and the current status of each.

AreaControlStatus
XSS Every user-facing value is passed through esc() before being written into HTML — prevents injected code from running ✓ PASS
CSP Content Security Policy restricts default-src 'none'. The only allowed outbound connection is to the Cloudflare Worker URL (for Card Maker). Every other external resource, frame, form, and object is blocked. ✓ PASS
SVG Diagrams from Claude are sanitized — scripts, event handlers, and external links are stripped before being embedded in the page ✓ PASS
Secrets No API keys or tokens in the code. All stored in GitHub Secrets and injected at runtime as environment variables ✓ PASS
Gmail scope App only has gmail.send permission — it cannot read, search, or modify your inbox under any circumstances ✓ PASS
Dependencies All packages locked to exact versions with SHA-256 hashes. Any tampered package will fail to install ✓ PASS
Actions GitHub Actions workflows use pinned commit SHAs instead of version tags — prevents supply chain attacks where a tag is moved to malicious code ✓ PASS
Feeds RSS feed content is sanitized and capped. URLs are validated to http/https only. Connection timeout prevents hung feeds from stalling the run ✓ PASS
Card Maker rate limit Card generation is capped at 10 per week via Cloudflare KV (server-side counter). Cannot be bypassed from the browser. Resets every Monday. Returns a clear message when the limit is reached. ✓ PASS
OAuth consent Google Cloud app is in "Testing" mode which causes the refresh token to expire every 7 days. Should be set to Production in Google Cloud Console ⚠ LOW
Overall verdict: PASS WITH NOTES. The app has strong security for a personal project. The one outstanding item is the OAuth consent screen — switching it from Testing to Production in Google Cloud Console prevents the refresh token from expiring every 7 days.

One Remaining Action Item

⚠️ Publish the OAuth Consent Screen

What it is: When you set up Gmail OAuth, Google puts new apps in "Testing" mode by default. In testing mode, refresh tokens expire every 7 days — meaning the email sending will silently fail after a week.

How to fix it:

  1. Go to console.cloud.google.com
  2. Select your project → APIs & Services → OAuth consent screen
  3. Click "Publish App" → Confirm

This makes the refresh token permanent so you never need to re-authorize.
