Three Microsoft Defender Zero-Days Under Active Exploitation — Two Remain Unpatched

Critical ⚠ Mitigation Only

Three zero-day privilege escalation vulnerabilities in Microsoft Defender — codenamed BlueHammer, RedSun, and UnDefend — are under active exploitation in the wild, as confirmed by Huntress. Only one of the three has been patched, leaving hundreds of millions of Windows systems exposed to attackers who can escalate from a low-privilege foothold to full SYSTEM-level control by abusing Defender's own elevated operations. The flaws were disclosed without coordinated notice by an independent researcher known as Chaotic Eclipse.

📅 Apr 17, 2026 9:21 AM EDT| Source: The Hacker News ↗
▶Attack Flow
Microsoft Defender Zero-Days (BlueHammer, RedSun, UnDefend) — Active Exploitation, Two Unpatched
Attackers exploit flaws in Microsoft Defender's privileged operations (file handling, service management, or input processing) to escalate from standard user-level access to SYSTEM privileges. Because Defender runs at the highest privilege level on Windows, successful exploitation effectively grants full control of the compromised host.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠You run Microsoft Defender as your primary or active endpoint protection on any Windows system
⚠Your environment includes Windows 10, Windows 11, or Windows Server machines with Defender enabled (the default configuration)
⚠You have users with any level of local access on Windows machines where Defender is active, including standard (non-admin) users
⚠Your environment relies on Defender in combination with other EDR tools but Defender is still running as an active service
✓You have completely disabled or uninstalled Microsoft Defender and use a third-party endpoint protection product exclusively with Defender services fully stopped
✓Your environment consists exclusively of non-Windows operating systems (Linux, macOS) without Defender for Endpoint deployed
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Monitor Microsoft Security Response Center (MSRC) for emergency out-of-band patches and apply any Defender definition/platform updates immediately as they are released — check via PowerShell: Get-MpComputerStatus | Select AntivirusSignatureVersion, AMProductVersion, AMEngineVersion.
  • 2Audit and restrict local user accounts and interactive logon rights on critical systems — attackers need an initial foothold before escalating, so reducing who can log on locally narrows the attack surface (use Group Policy: Computer Configuration > Windows Settings > Security Settings > Local Policies > User Rights Assignment > 'Allow log on locally').
  • 3Enable and enforce Attack Surface Reduction (ASR) rules via Microsoft Defender for Endpoint or Group Policy, particularly rules targeting process creation from Office apps, script execution, and credential theft — these limit attacker chains leading to exploitation: Set-MpPreference -AttackSurfaceReductionRules_Ids <GUID> -AttackSurfaceReductionRules_Actions Enabled.
  • 4Increase monitoring on all endpoints: enable PowerShell Script Block Logging (GPO: Administrative Templates > Windows Components > Windows PowerShell > Turn on Script Block Logging), Sysmon process creation logging, and Windows Security Event 4688 (Process Creation with command line) to detect exploitation attempts and post-exploitation behavior.
  • 5If operationally feasible for high-value assets, consider temporarily deploying an alternative real-time antivirus product and disabling Defender's real-time protection (Set-MpPreference -DisableRealtimeMonitoring $true) as a stopgap — only on systems where replacement coverage is confirmed active.
● Strategic This week & beyond
  • 1Implement a defense-in-depth endpoint strategy that does not rely solely on Microsoft Defender — layer a third-party EDR/XDR solution alongside or in place of Defender to ensure a single security product vulnerability cannot be the sole escalation path.
  • 2Deploy the principle of least privilege across all Windows endpoints using tiered administration models and Privileged Access Workstations (PAWs) so that even SYSTEM-level escalation on a single host cannot easily propagate to domain-wide compromise.
  • 3Establish a vulnerability response playbook specifically for security tooling zero-days that includes pre-approved fallback configurations, alternative AV deployment procedures, and communication templates — security software vulnerabilities require faster-than-normal response cycles.
  • 4Integrate continuous threat hunting for privilege escalation indicators into SOC operations, including baselining normal Defender service behavior (MsMpEng.exe, MpCmdRun.exe) and alerting on anomalous child process creation, file writes to sensitive directories, or token manipulation originating from Defender processes.
  • 5Engage with Microsoft's early notification programs (e.g., MAPP — Microsoft Active Protections Program) and threat intelligence sharing communities to receive advance warning of future Defender-related disclosures and exploits.
📚Concept BriefingClick a tag to expand
Privilege Escalation
A technique where an attacker upgrades their access level on a system — for example, going from a normal user to an administrator or SYSTEM account — gaining the ability to control everything on the machine.
All three Defender zero-days are privilege escalation flaws, allowing attackers to jump from a basic foothold to the highest Windows privilege level (SYSTEM) by exploiting Defender's own operations.
Zero-Day Vulnerability
A software flaw that is unknown to the vendor and has no patch available at the time of discovery or exploitation, leaving defenders with zero days of advance warning.
The researcher Chaotic Eclipse disclosed these three Defender flaws without coordinated notice to Microsoft, meaning they were weaponized before any patch existed — and two remain unpatched.
Security Software as Attack Surface
Security tools themselves can become targets because they run with elevated privileges and deep system access, meaning a flaw in the protector is often more dangerous than a flaw in what it protects.
Microsoft Defender runs as SYSTEM — the highest privilege on Windows — so exploiting it gives attackers more power than compromising almost any other application on the system.
Coordinated vs. Full Disclosure
Coordinated disclosure means a researcher privately notifies a vendor before going public, giving time to build a patch. Full (or uncoordinated) disclosure means publishing the flaw immediately, which can leave users exposed.
Chaotic Eclipse released these vulnerabilities as zero-days without advance notice to Microsoft, sparking controversy and leaving hundreds of millions of systems exposed before patches could be developed.
SYSTEM-Level Access (Windows)
SYSTEM is the most powerful built-in account on Windows, with unrestricted access to all files, processes, registry keys, and services — more powerful even than a standard administrator account.
Successful exploitation of these Defender flaws grants SYSTEM-level access, giving attackers the ability to fully control the compromised machine, disable security tools, and move laterally through the network.
🔍Threat Hunting Signals
High Anomalous child processes spawned by MsMpEng.exe or MpCmdRun.exe ►
Microsoft Defender's core engine process (MsMpEng.exe) and command-line utility (MpCmdRun.exe) should have a predictable set of child processes. Look for unexpected child processes such as cmd.exe, powershell.exe, whoami.exe, net.exe, or any unsigned binaries spawning from these parent processes, which may indicate exploitation of one of the three zero-days to execute code at SYSTEM level.
Sysmon Event ID 1 (Process Creation)Windows Security Event ID 4688EDR telemetry (process tree analysis)
High Defender service crashes, restarts, or unexpected configuration changes ►
Exploitation attempts may cause Defender's antimalware service (MsMpEng.exe) to crash or restart abnormally. Monitor for Windows Defender service stop/start events and for changes to Defender configuration keys in the registry (HKLM\SOFTWARE\Microsoft\Windows Defender) that are not initiated by authorized administrators or Group Policy.
Windows System Event Log (Event ID 7034, 7036, 7045)Windows Security Event ID 4657 (Registry value modification)Microsoft Defender operational log (Microsoft-Windows-Windows Defender/Operational)
Medium Unexpected token manipulation or privilege changes associated with Defender processes ►
Watch for Security events indicating token manipulation (such as impersonation or assignment of special privileges) involving Defender-related processes. Event ID 4672 (Special Privileges Assigned to New Logon) and Event ID 4673 (Sensitive Privilege Use) correlated with Defender process names may indicate privilege escalation attempts.
Windows Security Event ID 4672Windows Security Event ID 4673Sysmon Event ID 10 (Process Access)
Medium Suspicious file writes to Defender's program or data directories ►
Monitor for file creation or modification in Defender's installation directory (C:\ProgramData\Microsoft\Windows Defender, C:\Program Files\Windows Defender) by processes other than trusted Windows Update or Defender self-update mechanisms. Attackers may drop payloads or modify configuration files to exploit Defender's handling routines.
Sysmon Event ID 11 (File Create)Sysmon Event ID 2 (File Creation Time Changed)EDR file monitoring telemetry
High Rapid privilege escalation following initial low-privilege access ►
Correlate initial access indicators (e.g., new user logon via Event ID 4624 Type 10/3 with standard user privileges) with near-immediate SYSTEM-level process execution on the same host. A short time delta (seconds to minutes) between initial logon and SYSTEM-level command execution is consistent with automated exploit chain behavior observed in these campaigns.
