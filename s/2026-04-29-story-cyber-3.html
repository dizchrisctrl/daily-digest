DPRK Slips Malware Into npm by Tricking Claude Into Adding It as a Dependency

High ⚠ Mitigation Only

North Korean operators engineered an npm package, "@validate-sdk/v2", together with a supporting ecosystem of fake firms, seeded GitHub repos, and synthetic documentation, so that Anthropic's Claude Opus would recommend and add it as a project dependency. Once installed, the package's postinstall script pulls a DPRK-linked remote-access trojan. This is the first widely-reported case of a frontier LLM acting as the delivery vector in a software supply-chain attack.

📅 Apr 29, 2026 10:43 AM EDT| Source: The Hacker News ↗
▶Attack Flow
DPRK Weaponized Slopsquatting via Claude Opus (@validate-sdk/v2)
Attackers shaped the public training and retrieval signals (fake company sites, plausible READMEs, cross-referenced GitHub repos, stars, Stack Overflow-style mentions) that LLM coding assistants use to choose package recommendations, causing Claude Opus to insert the malicious npm package "@validate-sdk/v2" into developers' package.json. A postinstall hook then fetches and executes a RAT from attacker infrastructure during `npm install`.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your developers use Claude Opus or other LLM coding assistants to generate or edit package.json / dependency lists
⚠Your environment runs `npm install` with lifecycle scripts enabled (the default) on developer or CI machines
⚠Your environment has installed or referenced @validate-sdk/v2 in any project, lockfile, or container image since early 2026
✓You operate an internal package proxy (Artifactory/Verdaccio/Nexus) with an allowlist that blocks unvetted public packages
✓Your developers exclusively use packages from a curated, manually approved internal registry and `npm install --ignore-scripts` is enforced
⚠Your environment permits LLM agents to autonomously run `npm install` or commit dependency changes without human review
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Search all repos, lockfiles (package-lock.json, yarn.lock, pnpm-lock.yaml), Docker images, and CI caches for '@validate-sdk/v2' and remove it; treat any host that ran `npm install` against it as potentially compromised.
  • 2Rotate developer credentials, npm tokens, SSH keys, and cloud API keys from any workstation or CI runner that installed the package; review outbound network logs for unexpected connections during install windows.
  • 3Globally enforce `npm config set ignore-scripts true` on developer workstations and CI, and re-enable scripts only for explicitly vetted packages.
  • 4Configure your internal registry proxy to block @validate-sdk/* and require a human-approved allowlist for any new dependency suggested by an LLM agent.
  • 5Notify developers that LLM-suggested dependencies must be verified against the official npm registry, publisher history, and download trends before merging.
● Strategic This week & beyond
  • 1Insert a policy gate between LLM coding agents and package.json changes — for example, a pre-commit or PR check that flags any newly added dependency for human review with publisher age, download counts, and provenance metadata.
  • 2Adopt SLSA-style provenance and npm package signing requirements; prefer packages with verified publisher attestations and reject unsigned or low-reputation publishers in CI.
  • 3Stand up an internal mirror/proxy (Artifactory, Verdaccio, JFrog) with an allowlist model so that only curated package versions are reachable from developer and CI environments.
  • 4Add LLM-supply-chain scenarios to threat models and red-team exercises: assume adversaries are actively shaping the data your models retrieve, and test agentic workflows against poisoned-recommendation scenarios.
  • 5Deploy runtime protection on developer endpoints (EDR with install-time telemetry) and on CI runners so postinstall script behavior — outbound connections, child processes, file drops — is logged and alertable.
📚Concept BriefingClick a tag to expand
Slopsquatting
A supply-chain attack class where adversaries register package names that LLMs are likely to hallucinate or recommend, so that AI-suggested installs land on attacker-controlled code.
This incident is the weaponized form: rather than waiting for hallucinations, DPRK operators engineered the ecosystem signals that drive LLM recommendations toward their package.
LLM Data Poisoning
Manipulating the public content (websites, repos, forums, docs) that models train on or retrieve at runtime, in order to bias their outputs.
Fake firms, seeded GitHub stars, and synthetic documentation were planted specifically so Claude Opus would surface @validate-sdk/v2 as a credible answer.
npm Postinstall Abuse
The npm package lifecycle runs scripts (preinstall, postinstall) automatically during `npm install`, giving any dependency immediate code execution on the installing machine.
This is the execution primitive that turns a 'recommended' package into an instant RAT deployment with no further user action.
Agentic Supply Chain Risk
The risk that AI coding agents — which write package.json, run installs, and commit code — become an autonomous channel for introducing malicious dependencies at scale.
Traditional package vetting tools sit between humans and registries; almost none sit between an LLM agent and the package.json it is writing.
DPRK IT/Dev Operations
North Korean state-aligned campaigns that target developers and software supply chains for revenue generation and intelligence, often via fake personas and fake companies.
The fake-firm infrastructure and RAT tooling tie this campaign to the broader pattern of DPRK developer-targeted operations such as Contagious Interview.
