FBI and Indonesian Police Dismantle W3LL Phishing-as-a-Service Empire Behind $20M in Fraud

High ⚠ Mitigation Only

A joint FBI and Indonesian police operation dismantled the W3LL phishing-as-a-service (PhaaS) infrastructure, arrested its alleged developer in Indonesia, and disrupted a criminal network responsible for compromising thousands of credentials and attempting over $20 million in fraud globally. W3LL was a polished, subscription-based phishing toolkit that lowered the barrier for cybercriminals to launch sophisticated credential-harvesting campaigns against Microsoft 365 and other platforms, complete with adversary-in-the-middle MFA bypass capabilities.

📅 Apr 13, 2026 10:46 AM EDT| Source: The Hacker News ↗
▶Attack Flow
W3LL Phishing-as-a-Service Empire Takedown by FBI and Indonesian Police
W3LL provided turnkey phishing page templates cloning legitimate login portals (primarily Microsoft 365), combined with real-time adversary-in-the-middle session relay to intercept multi-factor authentication tokens. Operators purchased subscriptions, configured target lists, sent lure emails, and harvested credentials and session cookies through centralized admin panels — requiring no custom development.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your organization uses Microsoft 365 or Google Workspace with password-based or SMS/TOTP MFA authentication
⚠Your users receive external email and may click links to login portals
⚠Your organization lacks phishing-resistant MFA such as FIDO2 security keys or passkeys
⚠Your environment does not monitor for anomalous sign-in locations, impossible travel, or session token anomalies
✓Your organization has fully deployed FIDO2/passkeys as the sole authentication method with no password fallback
✓Your environment blocks all external email or operates entirely air-gapped from the internet
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Review Azure AD / Entra ID and Google Workspace sign-in logs for the past 90 days for anomalous sign-ins: impossible travel, unfamiliar IP addresses, or new device registrations that preceded mailbox rule changes or data exfiltration.
  • 2Revoke active sessions and force password resets for any accounts showing suspicious sign-in patterns — in Entra ID use 'Revoke Sessions' in the user blade; in Google Admin use 'Sign out user from all sessions'.
  • 3Enable Conditional Access policies (Azure AD) or Context-Aware Access (Google) to block sign-ins from unmanaged devices, untrusted locations, and anonymous VPN/proxy IPs immediately.
  • 4Audit mailbox rules in Exchange Online (Get-InboxRule) and Gmail (Admin Console > Email log search) for auto-forwarding rules or delete rules that attackers commonly create post-compromise.
  • 5Alert your user base about the W3LL takedown and the continued risk from copycat PhaaS platforms — remind them to verify login page URLs and report suspicious emails immediately.
● Strategic This week & beyond
  • 1Adopt FIDO2 security keys or platform passkeys as the primary authentication method across all cloud identity providers; this is the single most effective countermeasure against AiTM phishing toolkits like W3LL, as cryptographic origin-binding prevents credential relay.
  • 2Implement token protection / token binding (Entra ID Conditional Access Token Protection, currently in preview) to bind session tokens to specific devices, rendering stolen session cookies unusable.
  • 3Deploy a phishing-resistant email security gateway with real-time URL sandboxing and brand impersonation detection to catch W3LL-style cloned login pages before they reach inboxes.
  • 4Establish continuous session monitoring and anomaly detection using UEBA (User and Entity Behavior Analytics) to flag session hijacking indicators such as sudden geo-shifts within an active session.
  • 5Conduct regular tabletop exercises and red team assessments specifically simulating AiTM phishing scenarios to validate detection and response capabilities.
📚Concept BriefingClick a tag to expand
Phishing-as-a-Service
A criminal business model where a developer builds and sells turnkey phishing toolkits — including fake login pages, credential harvesting panels, and hosting — to other criminals on a subscription basis.
W3LL was a fully realized PhaaS platform with customer support and update cycles, enabling thousands of credential thefts and over $20M in fraud by lowering the technical barrier for phishing attacks.
Adversary-in-the-Middle (AiTM)
An attack technique where the attacker positions themselves between the victim and the legitimate service, relaying authentication in real time to capture session tokens and bypass multi-factor authentication.
W3LL's toolkit used AiTM relaying to intercept MFA tokens in real time, rendering standard SMS and TOTP-based MFA ineffective against its phishing campaigns.
FIDO2 / Passkeys
A phishing-resistant authentication standard that uses public-key cryptography bound to the specific website origin, making it impossible for attackers to relay credentials to a fake site.
FIDO2/passkeys are the primary technical countermeasure that would have neutralized W3LL's AiTM attacks, since cryptographic origin verification prevents credential relay to cloned portals.
Supply-Side Disruption
A law enforcement strategy that targets the developers and infrastructure providers enabling cybercrime, rather than pursuing individual end-user criminals one by one.
The FBI-Indonesian operation targeted the W3LL platform developer rather than individual phishing operators, aiming to disrupt the supply chain that enabled thousands of downstream attacks.
Credential Harvesting
The systematic collection of usernames, passwords, and session tokens from victims, typically through fake login pages or malware, for use in account takeover or fraud.
W3LL's core function was credential harvesting at scale — its phishing pages captured login data and session cookies from thousands of victims across Microsoft 365 and other platforms.
🔍Threat Hunting Signals
High Anomalous sign-in with immediate mailbox rule creation ►
Look for sign-in events from unusual IPs or geolocations (especially VPN/proxy exit nodes) immediately followed by creation of inbox rules that forward, redirect, or delete emails. This is a hallmark post-compromise action by W3LL operators to maintain access and intercept password reset notifications. Time delta of < 5 minutes between sign-in and rule creation is highly suspicious.
Azure AD / Entra ID Sign-in LogsExchange Online Unified Audit Log (Set-InboxRule, New-InboxRule)Google Workspace Admin Audit Log
High Impossible travel or rapid geo-shift in authenticated sessions ►
Detect scenarios where the same user account authenticates from geographically distant locations within an implausibly short time window (e.g., two countries within minutes). AiTM attacks cause the legitimate user's session to originate from their real location while the attacker's replayed session originates from a different location simultaneously.
Azure AD / Entra ID Sign-in Logs (Risk Detections)Google Workspace Login Audit EventsSIEM correlation rules
High Session token reuse from a new device or IP ►
Monitor for existing session tokens being presented from device fingerprints or IP addresses that differ from the original authentication event. W3LL's AiTM relay captures session cookies which are then replayed from attacker infrastructure. Look for UserAgent or device ID mismatches within the same session.
Azure AD / Entra ID Sign-in Logs (Token fields)Cloud Access Security Broker (CASB) logsWeb application firewall logs
Medium Spike in OAuth app consent or app registrations ►
After credential compromise, attackers may register OAuth applications or grant consent to malicious apps to maintain persistent access beyond the stolen session. Look for unusual OAuth consent grants, especially to apps with broad permissions (Mail.Read, Mail.Send, Files.ReadWrite.All).
Azure AD / Entra ID Audit Logs (Consent to application)Google Workspace Admin Console (Third-party app access)Microsoft Defender for Cloud Apps
Medium Phishing page infrastructure indicators in proxy/DNS logs ►
Search DNS query logs and web proxy logs for recently registered domains mimicking Microsoft 365 or Google login pages. Look for domains using typosquatting, homoglyph characters, or subdomains containing 'login', 'signin', 'outlook', or 'microsoftonline'. Even with W3LL's infrastructure taken down, copycat platforms may reuse similar domain patterns.
DNS query logs (passive DNS)Web proxy / Secure Web Gateway logsEmail gateway URL reputation logs
