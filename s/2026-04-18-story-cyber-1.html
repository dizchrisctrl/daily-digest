⚠ Vulnerability Advisory

Critical Thymeleaf SSTI Flaw Lets Attackers Bypass Sandbox and Execute Code on Servers

CVSS
9.1
Critical ✓ Patch Available

A critical Server-Side Template Injection (SSTI) vulnerability in the Thymeleaf Java template engine allows unauthenticated attackers to bypass the expression evaluation sandbox and achieve full remote code execution on affected servers. With a CVSS score of 9.1, any Java/Spring application that passes user-controlled input into Thymeleaf template resolution—even indirectly—is at risk of complete server compromise. Given Thymeleaf's widespread adoption as the default template engine in Spring Boot, the blast radius is significant across enterprise and SaaS environments.

📅 Apr 17, 2026 7:06 PM EDT| Source: CSO Online ↗ |CVE-2026-40478
▶Attack Flow
CVE-2026-40478 – Thymeleaf SSTI Sandbox Bypass (RCE)
An unauthenticated attacker sends a crafted HTTP request containing a malicious Thymeleaf expression payload. When the application incorporates user input into template resolution (e.g., via view name manipulation or direct template string concatenation), the payload exploits a gap in Thymeleaf's sandbox filtering logic, bypassing expression restrictions to invoke arbitrary Java methods and execute operating system commands on the server.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your application uses Thymeleaf as a server-side template engine (check for thymeleaf-spring* or thymeleaf-core in your dependency tree)
⚠Your application passes any user-controlled input into Thymeleaf template names, fragment expressions, or template strings (e.g., dynamic view name resolution from request parameters)
⚠Your Spring Boot application uses Thymeleaf as the default view resolver and exposes endpoints that accept user input influencing view rendering
✓Your application uses Thymeleaf only with fully static templates where no user input is ever interpolated into template resolution logic
✓You have already updated Thymeleaf to the patched version released on April 17, 2026
✓Your Java web applications use a different template engine (e.g., FreeMarker, JSP, Mustache) and do not include Thymeleaf in their dependency tree
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Inventory all applications with Thymeleaf in their dependency tree: run `mvn dependency:tree | grep thymeleaf` (Maven) or `gradle dependencies | grep thymeleaf` (Gradle) across all Java projects.
  • 2Update Thymeleaf to the patched version immediately by modifying your pom.xml or build.gradle dependency declarations. If using Spring Boot's dependency management, override the Thymeleaf version in your BOM or properties (e.g., `<thymeleaf.version>PATCHED_VERSION</thymeleaf.version>`).
  • 3If immediate patching is not possible, implement a WAF rule to block HTTP requests containing Thymeleaf expression patterns (e.g., `__${`, `__#{`, `*{`, `${T(`) in URL paths, query parameters, and POST bodies as a temporary mitigation.
  • 4Audit application code for any locations where user input is concatenated into Thymeleaf view names or template strings. Refactor these to use safe parameterized data binding (model attributes) instead.
  • 5Review web server and application logs for unusual patterns in URL paths or parameters containing Thymeleaf expression syntax (e.g., `__${`, `java.lang.Runtime`, `T(java.lang`) that may indicate active exploitation attempts.
● Strategic This week & beyond
  • 1Adopt Software Composition Analysis (SCA) tooling (e.g., Dependabot, Snyk, OWASP Dependency-Check) in CI/CD pipelines to receive automated alerts when critical vulnerabilities are disclosed in transitive dependencies like Thymeleaf.
  • 2Establish a secure coding standard that prohibits concatenation of user input into template names or template strings; enforce this via static analysis rules (e.g., Semgrep or CodeQL custom queries for SSTI patterns).
  • 3Deploy a Web Application Firewall (WAF) with regularly updated rule sets that detect template injection payloads across all public-facing Java applications.
  • 4Implement application-level sandboxing and least-privilege principles: run Java application processes with minimal OS permissions so that even successful RCE limits lateral movement and data access.
  • 5Conduct periodic penetration testing specifically targeting SSTI and expression language injection across all Java web applications in your portfolio.
📚Concept BriefingClick a tag to expand
Server-Side Template Injection
An attack where user-supplied input is interpreted as executable template code by the server's template engine, allowing the attacker to run arbitrary commands on the server.
CVE-2026-40478 is an SSTI vulnerability—attackers craft input that Thymeleaf's expression engine evaluates as code, bypassing the sandbox to achieve RCE.
Expression Language Sandbox
A security mechanism built into template engines that restricts which expressions, methods, and objects can be accessed during template evaluation, aiming to prevent code execution.
The core of this vulnerability is a bypass of Thymeleaf's sandbox—the filtering logic that was supposed to block dangerous method calls had a critical gap.
Remote Code Execution
A class of vulnerability that allows an attacker to run arbitrary commands or code on a target system remotely, typically leading to full system compromise.
The sandbox bypass in Thymeleaf escalates a template injection from data leakage to full RCE, allowing attackers to execute OS commands on the server.
Dependency Supply Chain Risk
The security risk introduced by third-party libraries and frameworks included in your application, where a vulnerability in any dependency becomes your vulnerability.
Thymeleaf is pulled in as a transitive dependency by Spring Boot—many teams may not realize they are running a vulnerable version until they audit their dependency tree.
CVSS Scoring
The Common Vulnerability Scoring System is an industry-standard framework that assigns a numerical severity score (0–10) to vulnerabilities based on factors like attack complexity and impact.
CVE-2026-40478's CVSS 9.1 score reflects that exploitation is network-based, requires no authentication, and results in high confidentiality and integrity impact.
🔍Threat Hunting Signals
High Thymeleaf expression patterns in HTTP request data ►
Look for Thymeleaf-specific expression syntax in URL paths, query parameters, headers, and POST bodies—patterns such as `__${`, `__#{`, `${T(java.lang`, `${#rt`, `Runtime.getRuntime().exec(`, or `ProcessBuilder`. These indicate an attacker probing for or actively exploiting SSTI. Focus on any parameter value containing nested expressions or Java class references.
WAF logsWeb server access logs (Apache/Nginx)Application server request logs (Tomcat catalina.out, Spring Boot request logging)
High Unexpected child processes spawned by Java application server ►
Monitor for the Java process (e.g., java, tomcat) spawning shell processes (bash, sh, cmd.exe, powershell.exe) or utility binaries (curl, wget, whoami, id, cat). This is a strong indicator that RCE has been achieved through SSTI or another code execution vector. Correlate the timestamp with incoming HTTP requests.

Medium Anomalous outbound network connections from Java application servers ►
After achieving RCE, attackers commonly initiate reverse shells or download second-stage payloads. Watch for the Java process or its child processes making outbound connections to unusual IP addresses, non-standard ports, or known C2 infrastructure. Baseline normal outbound traffic from application servers to detect deviations.

Search application logs for exceptions containing class names like `ognl.`, `T(java.lang.Runtime)`, `SpringEL`, or unusual expression strings. A spike in these errors can indicate active scanning.
