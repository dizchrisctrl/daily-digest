Bitwarden CLI Trojanized via Poisoned GitHub Action

Critical ⚠ Mitigation Only

TeamPCP, a threat group tracked by Socket.dev, compromised a GitHub Action used inside Bitwarden's CI/CD release pipeline and used that foothold to publish a trojanized build of Bitwarden CLI (@bitwarden/cli) through official npm channels. Because the malicious binary was produced, signed, and published by the legitimate pipeline, users who installed or updated the CLI during the compromise window received attacker-modified code that inherited full trust from the Bitwarden brand and signature chain.

📅 Apr 23, 2026 7:09 PM EDT| Source: CSO Online ↗
▶Attack Flow
TeamPCP — Bitwarden CLI Supply Chain Compromise via Poisoned GitHub Action
A poisoned third-party GitHub Action executed inside Bitwarden's release workflow with access to build secrets and publish tokens, allowing TeamPCP to inject malicious code into the Bitwarden CLI build artifacts before they were signed and pushed to npm. End users installing or updating @bitwarden/cli during the window unknowingly pulled a backdoored password manager binary from the official registry.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠You installed or updated @bitwarden/cli via npm during April 2026
⚠Your CI/CD jobs invoke bw (Bitwarden CLI) to fetch secrets at build or deploy time
⚠Your developers use Bitwarden CLI locally with an unlocked session to access production credentials
✓Your environment only uses the Bitwarden desktop app, browser extension, or mobile app and never installed the CLI package
✓You pin @bitwarden/cli to a specific pre-April 2026 version and verified hashes against Bitwarden's published checksums
✓Your environment uses self-hosted Bitwarden and never pulled the npm CLI distribution
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Inventory all hosts, developer laptops, and CI runners with @bitwarden/cli installed (npm ls -g @bitwarden/cli; which bw) and identify install/update timestamps in April 2026.
  • 2Uninstall suspect versions immediately (npm uninstall -g @bitwarden/cli) and block the package in internal registries/proxies until Bitwarden publishes a vetted clean version and checksum.
  • 3Treat any Bitwarden vault accessed via the CLI during the window as compromised: rotate master passwords, revoke API keys, and rotate every secret the CLI could read.
  • 4Rotate CI/CD secrets and tokens (cloud keys, deploy tokens, signing keys) that were retrievable via bw get in affected pipelines within the exposure window.
  • 5Review outbound network logs from hosts running the trojanized CLI for unexpected HTTPS destinations and pull Bitwarden session activity from server-side audit logs.
● Strategic This week & beyond
  • 1Pin all GitHub Actions to immutable commit SHAs rather than mutable tags (uses: org/action@<sha>) and enforce this via repository rulesets or policy-as-code.
  • 2Adopt ephemeral, least-privilege CI runners with scoped OIDC-issued tokens instead of long-lived publish credentials, and add egress allowlists on runners.
  • 3Require reproducible builds plus independent verification (e.g., SLSA Level 3+, signed provenance attestations) before promoting artifacts to public registries.
  • 4Move high-value secret retrieval out of interactive CLIs and into purpose-built secret brokers with short-lived credentials and per-request audit logging.
  • 5Subscribe to supply chain monitoring (Socket, OSV, npm audit signals) and maintain a documented playbook to quarantine and rebuild from a compromised upstream dependency.
📚Concept BriefingClick a tag to expand
Supply Chain Attack
An attack that compromises a trusted upstream component — a library, build tool, or vendor — so that downstream consumers receive malicious code through legitimate distribution channels.
The Bitwarden CLI users received a trojanized but properly signed binary through npm; the compromise happened upstream in the build pipeline, not at the user's endpoint.
CI/CD Pipeline Compromise
Subverting the automated build, test, and release system so that attacker code is produced alongside — or inside — legitimate artifacts using the project's own signing and publishing privileges.
A poisoned GitHub Action ran inside Bitwarden's release workflow with the same trust as maintainer-authored code, letting attackers publish as Bitwarden.
Action Pinning
Referencing GitHub Actions by immutable commit SHA rather than by mutable tags or branches, so an attacker who pushes a new tag cannot silently change what runs in your pipeline.
If the compromised action had been SHA-pinned, a maintainer-token takeover or malicious retag would not have flowed into the Bitwarden build.
Trusted Publisher Model
The assumption that packages from a well-known org on an official registry are safe because the publisher is reputable and the artifact is signed.
This incident shows the model's blind spot: a signed release from Bitwarden's own pipeline is still malicious if the pipeline itself was subverted.
Secret Sprawl
The proliferation of long-lived credentials, API keys, and tokens across developer machines, scripts, and CI jobs, making blast radius huge when any one host is compromised.
Bitwarden CLI is typically installed exactly where secrets are densest, so a single trojanized copy can harvest dozens of high-value credentials in one sweep.
