⚠ Vulnerability Advisory

Critical RCE Flaw in protobuf.js Puts Millions of JavaScript Apps at Risk

CVSS
9.8
Critical ✓ Patch Available

A critical remote code execution vulnerability has been discovered in protobuf.js, the most popular JavaScript implementation of Google's Protocol Buffers with tens of millions of monthly npm downloads. The flaw allows an attacker to craft a malicious protobuf message that exploits improper input sanitization during deserialization to pollute JavaScript's Object.prototype, escalating to arbitrary code execution on the server. A public proof-of-concept exploit is now circulating, making exploitation trivial against any unpatched system that deserializes untrusted protobuf messages.

📅 Apr 18, 2026 11:09 AM EDT| Source: BleepingComputer ↗
▶Attack Flow
Critical RCE via Prototype Pollution in protobuf.js (protobufjs npm package)
An attacker sends a specially crafted Protocol Buffers message to a vulnerable Node.js service using protobuf.js. The library's deserialization logic fails to sanitize keys like `__proto__` or `constructor.prototype`, allowing the attacker to inject properties onto JavaScript's root Object.prototype. This prototype pollution is then escalated to full remote code execution when the polluted properties interact with downstream code that evaluates dynamic properties or constructs commands.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your Node.js applications or services use the protobufjs npm package directly as a dependency
⚠Your environment runs gRPC-based Node.js microservices (which commonly depend on protobuf.js transitively via @grpc/grpc-js or similar)
⚠Your services deserialize protobuf messages from untrusted sources such as external clients, partner APIs, or public endpoints
⚠Your application has protobuf.js deep in its transitive dependency tree even if not directly imported (check with `npm ls protobufjs`)
✓You use protobuf implementations in languages other than JavaScript (e.g., Go, Python, Java, Rust protobuf libraries)
✓Your Node.js services only deserialize protobuf messages from fully trusted, authenticated internal sources with no external input path
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Run `npm ls protobufjs` and `yarn why protobufjs` across all projects and CI/CD pipelines to identify every direct and transitive instance of the vulnerable package.
  • 2Update protobufjs to the latest patched version immediately using `npm update protobufjs` or by pinning the fixed version in package.json; for transitive dependencies, use npm overrides or yarn resolutions to force the patched version (e.g., add `"overrides": {"protobufjs": ">=FIXED_VERSION"}` to package.json).
  • 3If immediate patching is not possible, implement input validation at the network boundary to reject protobuf messages containing fields with keys like `__proto__`, `constructor`, or `prototype` before they reach deserialization logic.
  • 4Deploy Object.prototype freeze as a defense-in-depth measure where feasible by adding `Object.freeze(Object.prototype)` early in your application bootstrap — test thoroughly as this can break some libraries.
  • 5Monitor application logs and WAF/API gateway logs for anomalous protobuf payloads, unexpected server errors, or signs of code execution (e.g., unexpected child process spawns from Node.js workers).
● Strategic This week & beyond
  • 1Implement Software Composition Analysis (SCA) tooling (e.g., Snyk, Socket.dev, npm audit) in CI/CD pipelines to continuously monitor for vulnerabilities in direct and transitive dependencies and block deployments of unpatched packages.
  • 2Adopt a dependency review and pinning strategy that includes regular audits of deeply nested transitive dependencies, not just direct imports, with automated alerting for critical-severity advisories.
  • 3Evaluate replacing protobuf.js with alternative protobuf implementations or consider using protobuf deserialization in a sandboxed or isolated context (e.g., worker threads with restricted permissions) to limit the blast radius of future deserialization vulnerabilities.
  • 4Implement runtime application self-protection (RASP) or Node.js-specific security hardening such as the --frozen-intrinsics flag (where supported) to prevent prototype pollution from affecting built-in objects.
  • 5Establish a microservice input validation layer that enforces schema-strict deserialization with allowlisted field names at API gateways before messages reach application-level protobuf parsing.
📚Concept BriefingClick a tag to expand
Prototype Pollution
A class of JavaScript vulnerability where an attacker injects properties into Object.prototype, the root object from which all JavaScript objects inherit, thereby altering the behavior of the entire application.
This is the core vulnerability mechanism in protobuf.js — the deserialization logic fails to block dangerous keys like __proto__, allowing attackers to pollute the shared prototype and escalate to code execution.
Remote Code Execution
A vulnerability that allows an attacker to run arbitrary code on a target system remotely, typically over a network, without needing prior access.
The prototype pollution in protobuf.js is escalated to full RCE, meaning an attacker who can send a crafted network message gains complete control over the server process.
Supply Chain Dependency Risk
The security risk that arises when a vulnerability in a widely-used library propagates to all applications that depend on it, directly or transitively.
protobuf.js has tens of millions of monthly downloads and sits deep in dependency trees of countless applications, making a single flaw in this package a systemic risk to the entire Node.js ecosystem.
Protocol Buffers Deserialization
The process of converting serialized Protocol Buffers binary data back into structured in-memory objects, which is a common attack surface when handling untrusted input.
The vulnerability is triggered during deserialization of crafted protobuf messages — any service that deserializes protobuf from untrusted sources is directly exposed to this attack.
Transitive Dependencies
Libraries that your application depends on indirectly — they are dependencies of your dependencies, often buried several layers deep in the dependency tree.
Many applications are vulnerable not because they directly import protobuf.js, but because it is pulled in transitively by gRPC libraries and other packages, making discovery and patching more complex.
🔍Threat Hunting Signals
High Anomalous protobuf message fields containing prototype pollution keys ►
Look for inbound network traffic or deserialized protobuf payloads containing field names such as `__proto__`, `constructor`, `prototype`, or `constructor.prototype`. These keys should never appear in legitimate protobuf schemas and indicate a prototype pollution attempt. Inspect API gateway logs, WAF logs, and application-level request logging for these patterns.
API gateway/WAF logsApplication request logsNetwork traffic captures (PCAP/IDS)
High Unexpected child process spawns from Node.js processes ►
Monitor for Node.js worker or server processes spawning unexpected child processes (e.g., /bin/sh, cmd.exe, curl, wget). Successful prototype pollution to RCE exploitation typically results in arbitrary command execution, which manifests as unusual child process creation from the Node.js runtime. Check process auditing and EDR telemetry.
EDR/endpoint telemetryLinux auditd logsProcess monitoring (sysmon for Windows)
Medium Sudden increase in application errors or crashes in protobuf deserialization paths ►
A spike in unhandled exceptions, type errors, or unexpected property access errors in code paths that deserialize protobuf messages may indicate prototype pollution attempts that are altering object behavior. Review application error logs and APM dashboards for anomalous error patterns in protobuf-related modules.
Application error logs (stdout/stderr)APM tools (Datadog, New Relic, etc.)Node.js uncaughtException/unhandledRejection handlers
High Outbound network connections from services that normally only receive inbound traffic ►
After achieving RCE, attackers commonly establish reverse shells or exfiltrate data. Monitor for unexpected outbound connections from Node.js service containers or processes, especially to unfamiliar IPs or on unusual ports. Focus on services that handle protobuf deserialization.
