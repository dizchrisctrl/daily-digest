China-Linked Hackers Use SMTP Port 25 as Covert Channel to Steal Cloud Credentials at Scale

Critical ⚠ Mitigation Only

A China-aligned threat actor is deploying a Linux ELF backdoor that uses SMTP port 25 as a covert command-and-control channel to harvest credentials from AWS, GCP, Azure, and Alibaba Cloud workloads at scale. The C2 infrastructure employs a selective handshake mechanism that makes it invisible to internet-wide scanning tools like Shodan and Censys, representing a significant blind spot for defenders relying on conventional threat intelligence. A single compromised workload can cascade into full account takeover across multiple cloud providers.

📅 Apr 14, 2026 8:12 AM EDT| Source: CSO Online ↗
▶Attack Flow
China-Linked SMTP Port 25 Covert C2 Cloud Credential Theft Campaign
The attacker deploys a Linux ELF backdoor on cloud workloads that queries cloud Instance Metadata Service (IMDS) endpoints to harvest temporary credentials and tokens from AWS, GCP, Azure, and Alibaba Cloud. Stolen credentials are exfiltrated over SMTP port 25 using a covert C2 channel that embeds encoded data within SMTP protocol envelope fields, with a selective handshake mechanism that only responds to implants presenting the correct protocol-level signal sequence—rendering the C2 server invisible to all other probes.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠You run Linux-based workloads on AWS, GCP, Azure, or Alibaba Cloud
⚠Your cloud instances use IMDSv1 (AWS) or have unrestricted metadata service access
⚠Your network firewalls allow outbound SMTP traffic on port 25 from non-mail-server workloads
⚠Your environment lacks behavioral anomaly detection for outbound SMTP traffic from cloud instances
✓You have enforced IMDSv2 on all AWS instances and equivalent metadata hardening on other clouds, and block outbound port 25 from non-mail workloads
✓Your cloud workloads run exclusively on Windows operating systems with no Linux instances
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Audit outbound port 25 traffic: Immediately review VPC flow logs (AWS), VPC Flow Logs (GCP), and NSG flow logs (Azure) for any non-mail-server instances generating outbound SMTP connections on port 25. Block outbound port 25 from all workloads that are not designated mail servers.
  • 2Enforce IMDSv2 on all AWS instances: Run `aws ec2 modify-instance-metadata-options --instance-id <id> --http-tokens required --http-put-response-hop-limit 1` for every EC2 instance. Apply equivalent metadata hardening on GCP (metadata concealment), Azure (restrict IMDS), and Alibaba Cloud.
  • 3Scan Linux workloads for unknown ELF binaries: Use host-based tools (e.g., `find / -type f -executable -newer /etc/hostname -exec file {} \; | grep ELF`) to identify recently deployed ELF executables. Cross-reference against known software inventories.
  • 4Rotate all cloud credentials and temporary tokens: Immediately rotate IAM access keys, service account keys, and any long-lived tokens on workloads suspected of compromise. Revoke active STS sessions on AWS using `aws iam put-role-policy` deny-all policies.
  • 5Review CloudTrail, GCP Audit Logs, and Azure Activity Logs for anomalous credential usage: Look for API calls from unexpected source IPs, unusual services accessed, or cross-region activity that may indicate stolen credential use.
● Strategic This week & beyond
  • 1Implement network-layer behavioral anomaly detection: Deploy NDR (Network Detection and Response) solutions that perform protocol-level inspection beyond simple port-based allowlisting, specifically monitoring for anomalous SMTP traffic patterns, unusual SMTP envelope fields, and SMTP connections from non-mail workloads.
  • 2Adopt a zero-trust architecture for cloud metadata access: Enforce the principle of least privilege for IMDS access using instance-level firewall rules (iptables), container network policies, or cloud-native controls to ensure only authorized processes can reach metadata endpoints.
  • 3Establish egress filtering as a default security control: Implement cloud-native egress controls (AWS Security Groups, GCP firewall rules, Azure NSGs) with an explicit deny-all outbound policy and allowlist only required destinations and protocols per workload role.
  • 4Deploy runtime protection on all cloud workloads: Use CWPP (Cloud Workload Protection Platform) agents or eBPF-based runtime security tools that detect unauthorized binary execution, process injection, and anomalous system calls on Linux instances.
  • 5Integrate SMTP and legacy protocol monitoring into your SOC playbooks: Update SIEM detection rules to alert on SMTP traffic from non-mail hosts, unusually large SMTP envelope data, and connections to SMTP servers not on your organization's approved relay list.
📚Concept BriefingClick a tag to expand
Covert C2 Channel
A method where attackers hide command-and-control communications inside traffic from legitimate, expected network protocols so that monitoring tools do not flag it as malicious.
This campaign tunnels C2 traffic and exfiltrated credentials through SMTP port 25, which firewalls and monitoring tools typically allow as normal email traffic, making the adversary's infrastructure invisible to conventional detection.
Cloud IMDS Abuse
The Instance Metadata Service is a cloud-provider endpoint (e.g., 169.254.169.254) that provides virtual machines with configuration data and temporary credentials; attackers can query it to steal those credentials.
The backdoor harvests temporary credentials and tokens from IMDS endpoints across AWS, GCP, Azure, and Alibaba Cloud, turning a single compromised workload into a pivot point for cross-cloud account takeover.
Protocol Abuse
The technique of using a legitimate network protocol in ways it was not designed for—such as embedding malicious payloads in protocol fields—to bypass security controls that only inspect traffic at the port or protocol level.
SMTP on port 25 is expected on most networks and rarely inspected for malware; the attacker exploits this assumption to move data past firewalls and evade deep packet inspection typically focused on spam rather than C2.
Traffic Signaling / Port Knocking
A technique where a server only activates or responds on a port after receiving a specific, secret sequence of network signals, keeping the service hidden from unauthorized scanners.
The C2 server uses a selective handshake validation mechanism that responds only to implants with the correct signal sequence, making it appear dead or generic to Shodan, Censys, and other scanning tools used for threat intelligence.
Multi-Cloud Credential Theft
An attack that targets credentials across multiple cloud service providers simultaneously, enabling lateral movement and privilege escalation that spans organizational cloud boundaries.
This campaign targets AWS, GCP, Azure, and Alibaba Cloud credentials from a single backdoor, meaning one compromised Linux workload can provide attackers access to resources across all of an organization's cloud providers.
🔍Threat Hunting Signals
High Outbound SMTP (port 25) connections from non-mail-server workloads ►
Look for cloud instances that are not designated mail servers initiating outbound TCP connections on port 25. In multi-cloud environments, no compute workload should typically generate outbound SMTP unless it is an explicitly provisioned mail relay. Any such traffic from application servers, containers, or general-purpose VMs is a strong indicator of covert C2 or data exfiltration via SMTP abuse.

Use host-based monitoring tools or iptables logging to detect curl, wget, or unknown ELF binaries accessing the metadata service, which may indicate credential harvesting activity.

Search for ELF executables that have appeared in /tmp, /var/tmp, /dev/shm, or user home directories within recent timeframes that do not match known software packages. Focus on binaries without package manager provenance (dpkg -S or rpm -qf returning no results) and those with stripped symbols or packed/obfuscated sections.

Inspect SMTP traffic for unusual characteristics: oversized or encoded EHLO/MAIL FROM/RCPT TO fields, atypical SMTP response codes, connections that do not follow standard SMTP conversation flow, or SMTP sessions carrying unusually large data payloads. These may indicate encoded C2 commands or exfiltrated credentials embedded in SMTP envelope fields.
