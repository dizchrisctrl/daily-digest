⚠ Vulnerability Advisory

PhantomRPC: Five Ways to Hijack a Service That Isn't There

CVSS
7.8
High ⚠ No Fix Yet

A security researcher disclosed five distinct local privilege escalation chains in Windows RPC that all exploit the same architectural flaw: when a privileged client makes a call to a service that isn't running, the OS routes the call based on endpoint name rather than verified service identity. Low-privileged code that squats or pre-registers the endpoint can impersonate the absent service and inherit SYSTEM-level trust. Microsoft considers the behavior by-design and has not released a patch.

📅 Apr 27, 2026 11:31 AM EDT| Source: darkreading ↗
▶Attack Flow
PhantomRPC: Architectural RPC Endpoint Confusion in Windows (Unpatched)
A low-privileged local attacker registers or squats an ALPC port or named pipe associated with an offline Windows service, then races to claim the endpoint before the legitimate service starts. When a privileged client calls the named endpoint, the RPC subsystem connects it to the attacker, who can steal impersonation tokens or coerce trusted operations to gain SYSTEM.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your environment runs any supported Windows client or server edition with default services configuration
⚠You allow non-administrative users to execute arbitrary code (interactive logons, RDP, Citrix, dev workstations, terminal servers)
⚠Your endpoints have demand-start or rarely-running services (Print Spooler, Storage Service, Fax, etc.) that low-privileged users can trigger
✓Your environment only runs hardened single-purpose appliances where no untrusted local code can execute
⚠You rely solely on EDR signature-based detection without behavioral monitoring of ALPC/named-pipe creation by non-service processes
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Inventory which services are set to Demand Start (sc.exe query state= all | findstr /i DEMAND) and disable any that are unused, especially Print Spooler on non-print hosts (Set-Service -Name Spooler -StartupType Disabled; Stop-Service Spooler).
  • 2Restrict interactive and remote interactive logons for non-administrative users on Tier-0 and Tier-1 systems (domain controllers, jump hosts, file/print servers) via Group Policy 'Deny log on locally' / 'Deny log on through Remote Desktop Services'.
  • 3Enable EDR rules or Sysmon Event ID 17/18 (Pipe Created/Connected) and ALPC port creation telemetry to flag non-service processes creating named pipes or ALPC ports matching known service endpoints.
  • 4Apply Microsoft's existing mitigations for known 'Potato'-class and PrintNightmare-class issues (disable Print Spooler where not needed, enforce SMB signing, set RestrictDriverInstallationToAdministrators=1) as defense in depth.
  • 5Subscribe to MSRC advisories and the researcher's disclosure thread for any out-of-band guidance or future patch announcement.
● Strategic This week & beyond
  • 1Adopt a tiered administration model so SYSTEM-level local privilege escalation on user-tier endpoints does not yield credentials usable against Tier-0 assets (LAPS, credential guard, protected users group).
  • 2Deploy application allowlisting (WDAC, AppLocker) to prevent arbitrary unsigned binaries from running in user context — most PhantomRPC exploitation requires an attacker-supplied binary.
  • 3Enable Credential Guard and LSA protection so token theft from privileged processes is blunted even if RPC endpoint hijacking succeeds.
  • 4Build behavioral detections around the architectural pattern (process creating an ALPC port or named pipe whose name matches an installed service it does not host) rather than waiting for IOCs of specific PoCs.
  • 5Engage Microsoft and industry peers to push for cryptographic or SID-bound endpoint identity in RPC (treating endpoint name + expected service SID as the trust anchor, not the name alone).
📚Concept BriefingClick a tag to expand
RPC / ALPC
Remote Procedure Call and Advanced Local Procedure Call are the primary inter-process communication mechanisms Windows services use to expose privileged functionality to local clients.
PhantomRPC lives entirely inside this layer — it abuses how RPC resolves named endpoints when the owning service is offline.
Endpoint Squatting
Registering a well-known IPC name (named pipe, ALPC port) before the legitimate owner does, so that callers connect to the squatter instead.
All five PhantomRPC variants depend on the attacker claiming or pre-registering an endpoint that an absent privileged service is expected to own.
Race Condition
A flaw where the outcome depends on the timing of two events; an attacker who reliably wins the race causes unintended behavior.
Demand-start service variants of PhantomRPC require the attacker to claim the endpoint between the moment a client triggers service start and the moment the real service binds.
Token Impersonation
A Windows mechanism where a server can act on behalf of a connecting client using the client's access token; abuse lets a low-privileged server steal a high-privileged caller's identity.
Once a privileged client connects to the attacker's squatted endpoint, several PhantomRPC chains capture and impersonate the caller's SYSTEM token.
Architectural / By-Design Flaw
A weakness that arises from foundational design choices rather than a discrete coding bug, often resistant to point-fix patches.
Microsoft has reportedly classified PhantomRPC as by-design, meaning defenders must rely on configuration and detection rather than a patch.
🔍Threat Hunting Signals
High Non-service process creating an ALPC port or named pipe matching a known Windows service endpoint ►
Hunt for processes running as standard users that create named pipes or ALPC ports whose names overlap with installed-but-stopped services (e.g., \\.\pipe\spoolss, \RPC Control\<service>). Legitimate ownership should be a service host process running as the expected service SID.

High Demand-start service triggered shortly after suspicious endpoint creation ►
Correlate user-context creation of a service-named endpoint with subsequent service start events for the same service within seconds — a hallmark of the demand-start race variant.
