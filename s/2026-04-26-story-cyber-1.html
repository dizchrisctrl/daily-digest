'fast16': The Lua-Based Sabotage Framework That Predated Stuxnet by Five Years

High ⚠ Mitigation Only

SentinelOne researchers disclosed 'fast16,' a previously undocumented Lua-based cyber-sabotage framework dating to 2005 that targeted high-precision engineering calculation software. Rather than attacking PLCs like Stuxnet would five years later, fast16 corrupted the numerical outputs of the design and calculation tools engineers trust — pushing the timeline of state-grade industrial sabotage back at least half a decade and shifting where defenders should be looking.

📅 Apr 25, 2026 5:26 AM EDT| Source: The Hacker News ↗
▶Attack Flow
fast16 — Pre-Stuxnet Lua-Based Engineering Sabotage Framework
fast16 embedded itself as Lua scripts/plugins inside high-precision engineering calculation software, intercepting computations at the application layer and silently altering numerical outputs so that flawed values were baked into designs while still appearing plausible to reviewers. Because the malicious logic lived in userland scripting interfaces, it bypassed firmware/PLC defenses entirely and corrupted the engineering trust chain upstream of any physical controller.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your environment runs CAD, FEA, simulation, or other high-precision engineering calculation software that supports Lua or similar embedded scripting extensions
⚠Your engineers install third-party plugins, macros, or scripts into engineering applications without code review or signing requirements
⚠Your design and manufacturing pipeline trusts a single computational source of truth and does not cross-validate critical calculations with an independent solver or hand-derivation
⚠Your organization designs, simulates, or manufactures safety-critical or strategically sensitive components (aerospace, nuclear, defense, precision machining)
✓Your engineering workstations are fully air-gapped, run only signed vendor binaries, and disable all user scripting and plugin loading
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Inventory all engineering applications in use (CAD, CAE, FEA, simulation, ballistics, tolerance) and enumerate their Lua/script/plugin directories — typically under %APPDATA%, the install path, or user 'scripts'/'macros' folders.
  • 2Hash and review every .lua, .luac, plugin, and macro file found; flag anything not shipped by the vendor or not explicitly authorized by engineering.
  • 3Disable user-loaded scripting extensions in engineering software where the workflow allows (most vendors expose a 'disable scripts' or 'safe mode' setting).
  • 4Spot-check recent critical design outputs by re-running the computation on an independent solver or, for key dimensions, by hand — look for small but consistent numerical drift.
  • 5Restrict write permissions on engineering application install and plugin directories to administrators only, and alert on modifications via EDR/file integrity monitoring.
● Strategic This week & beyond
  • 1Establish a 'computational trust chain': require dual-solver or independent verification for safety-critical calculations before they leave the design phase.
  • 2Treat engineering workstations as high-value OT-adjacent assets — segment them, apply application allowlisting (e.g., WDAC/AppLocker), and forbid arbitrary plugin installation.
  • 3Mandate code signing for all engineering plugins and Lua scripts; reject unsigned extensions at load time where the host application supports it.
  • 4Build a regression corpus of known-good reference calculations and run it periodically against production engineering tools to detect output tampering.
  • 5Extend threat modeling beyond PLC/OT networks to the design-phase software stack, and include engineering applications in red-team and supply-chain risk assessments.
📚Concept BriefingClick a tag to expand
Computational Sabotage
Attacks that silently alter the numerical outputs of trusted calculation or design software so that flawed values are baked into whatever is later built or manufactured.
fast16 is the earliest known example: rather than breaking machines, it made the engineering software lie about the math.
Living-off-the-Scripting-Engine
Abusing legitimate embedded scripting interfaces (Lua, Python, VBA, JavaScript) inside host applications to run malicious logic that looks like ordinary user automation.
fast16 rode Lua's role as a plugin layer in engineering tools, blending in with benign user scripts.
Cyber-Physical Sabotage
Operations that use digital intrusion to cause physical-world harm or degradation, whether by manipulating controllers, processes, or design data.
fast16 reframes the genre: cyber-physical effects can originate in the CAD office, not just on the plant floor.
Engineering Trust Chain
The implicit assumption that CAD, simulation, and calculation software produce correct results, which engineers, manufacturers, and regulators all rely on without re-derivation.
fast16's whole value proposition was breaking this assumption silently and durably.
Design-Phase Compromise
Inserting defects upstream of manufacturing — into drawings, tolerances, or simulation results — so that production faithfully builds something subtly wrong.
This is fast16's strategic innovation and is arguably stealthier and more devastating than runtime PLC attacks.
🔍Threat Hunting Signals
High Unexpected .lua / .luac files in engineering application directories ►
Enumerate plugin, macro, and script directories of installed CAD/FEA/simulation software. Compare hashes of all Lua files against vendor-shipped baselines. Investigate any file not present in the vendor manifest, especially those with recent modification timestamps or unusual entropy.
EDR file eventsFile integrity monitoring (FIM)Endpoint inventory / software bill of materials
Medium Engineering applications spawning unusual child processes or making outbound network connections ►
CAD and FEA tools rarely need to make network calls beyond licensing servers. Hunt for these processes initiating DNS lookups, HTTP(S) connections to non-vendor destinations, or spawning shells/scripting interpreters.
EDR process telemetrySysmon (Event IDs 1, 3, 22)DNS logsNetwork firewall logs
High Statistically anomalous drift in engineering calculation outputs ►
Maintain a reference set of canonical inputs and known-good outputs for critical computations. Periodically re-run them on production workstations and compare; small consistent biases (e.g., systematic millimeter-scale offsets, off-by-tolerance values) may indicate output tampering.
QA/validation pipelinesIndependent reference solversEngineering change management systems
Medium Modifications to engineering app install paths or plugin folders by non-admin users ►
Alert on writes to application installation directories, plugin/extension folders, and per-user script directories from processes other than the vendor installer or update service.
