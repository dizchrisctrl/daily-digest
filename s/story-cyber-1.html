Adobe Patches Zero-Day in Acrobat That Attackers Exploited for Four Months

Critical ✓ Patch Available

Adobe has patched an actively exploited zero-day vulnerability in Adobe Acrobat and Adobe Acrobat Reader that allowed attackers to achieve arbitrary code execution via specially crafted PDF files. The vulnerability was exploited in the wild for at least four months before a patch was released, exposing hundreds of millions of endpoints to compromise through routine document workflows.

📅 Apr 13, 2026 4:52 PM EDT| Source: darkreading ↗
▶Attack Flow
Adobe Acrobat & Reader Actively Exploited Zero-Day (April 2026)
Attackers crafted malicious PDF files that exploit a flaw in Adobe Acrobat and Reader's PDF parsing engine. When a victim opens the weaponized PDF — typically delivered as an email attachment — the exploit triggers code execution in the context of the reader process, potentially giving the attacker full control of the target machine with no additional user interaction beyond opening the file.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠You have Adobe Acrobat or Adobe Acrobat Reader installed on any endpoints in your environment
⚠Your users routinely receive and open PDF attachments from external sources via email
⚠Your environment has not yet applied the Adobe April 2026 security update to all installations of Acrobat and Reader
⚠Your email gateway does not perform sandboxed detonation or deep inspection of PDF attachments
✓You do not use Adobe Acrobat or Reader and instead rely exclusively on alternative PDF readers (e.g., browser-native PDF viewers, Foxit with current patches)
✓You have already applied the April 2026 Adobe security update across all endpoints and confirmed deployment via asset inventory
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Deploy the Adobe April 2026 security update immediately to all installations of Adobe Acrobat and Adobe Acrobat Reader. Use Adobe's Admin Console, SCCM, Intune, or your endpoint management platform to push the update: for Acrobat, run the latest installer from Adobe's security bulletin page or trigger 'Help > Check for Updates' on managed endpoints.
  • 2Until the patch is confirmed deployed on all endpoints, configure Adobe Acrobat and Reader to open PDFs in Protected View (Edit > Preferences > Security (Enhanced) > enable 'Files from potentially unsafe locations' or set to 'All files'). This sandboxes the rendering engine.
  • 3Instruct your email security/gateway team to quarantine or sandbox-detonate all inbound PDF attachments from external senders for the next 72 hours while patching is completed.
  • 4Audit your environment for any instances of Adobe Acrobat or Reader that may have been installed outside of IT management (shadow IT), particularly on developer workstations, executive laptops, and contractor machines.
  • 5Communicate to all users via security alert that they should not open unexpected or unsolicited PDF attachments until further notice, and report any suspicious PDFs to the SOC immediately.
● Strategic This week & beyond
  • 1Implement automated patch management for Adobe products with SLA targets of 24-48 hours for critical/zero-day patches, using tools like SCCM, Intune, WSUS (for MSI packages), or Adobe's AUSST (Adobe Update Server Setup Tool) for air-gapped environments.
  • 2Deploy or enhance email attachment sandboxing capabilities that detonate PDF files in an isolated environment before delivery to end-user mailboxes, flagging files that trigger anomalous process creation or network callbacks.
  • 3Evaluate migrating low-risk PDF viewing use cases to browser-native PDF viewers (Chrome, Edge built-in readers) which operate within a stricter sandbox and have a smaller attack surface than full Adobe Acrobat/Reader installations.
  • 4Implement application control policies (e.g., Microsoft Defender Application Control, AppLocker) that restrict child process creation from Adobe Acrobat and Reader processes — legitimate PDF viewing should not spawn cmd.exe, PowerShell, or other interpreters.
  • 5Conduct a retroactive threat hunt covering the past four months (December 2025 through April 2026) focused on anomalous process trees originating from Acrobat or AcroRd32 processes, as your environment may have been compromised during the pre-patch exploitation window.
📚Concept BriefingClick a tag to expand
Zero-Day Vulnerability
A software flaw that is unknown to the vendor and has no available patch, giving defenders 'zero days' to respond before exploitation begins.
This Adobe vulnerability was a zero-day exploited for approximately four months before Adobe became aware and released a fix, leaving all Acrobat/Reader users defenseless during that period.
Document-Based Exploitation
An attack technique where malicious code is embedded within common document formats (PDF, Office files) that executes when the document is opened by vulnerable software.
The attackers weaponized the universally trusted PDF format to deliver the exploit, requiring only that victims open what appeared to be a normal document.
Arbitrary Code Execution
A vulnerability impact where an attacker can run any code of their choosing on the victim's machine, often with the same privileges as the exploited application.
Successful exploitation of this Acrobat flaw gives the attacker the ability to execute commands on the victim's system, potentially leading to full compromise.
Protected View / Sandboxing
A security feature that opens files in a restricted environment with limited system access, preventing malicious content from reaching the full operating system.
Enabling Protected View in Adobe Acrobat/Reader can mitigate exploitation by restricting the code execution context of the PDF parsing engine.
Patch Dwell Time
The duration between when a vulnerability is first exploited in the wild and when a patch becomes available, during which all affected systems remain exposed.
The four-month dwell time for this zero-day means organizations were vulnerable from approximately December 2025 through April 2026, and retroactive investigation is essential.
🔍Threat Hunting Signals
High Anomalous child processes spawned by Adobe Acrobat or Reader ►
Look for process creation events where the parent process is Acrobat.exe, AcroRd32.exe, or AdobeAcrobat.exe and the child process is cmd.exe, powershell.exe, wscript.exe, cscript.exe, mshta.exe, rundll32.exe, or any unexpected binary. Legitimate PDF viewing should not spawn command interpreters or scripting engines.
EDR telemetry (CrowdStrike, Defender for Endpoint, SentinelOne)Windows Security Event Log (Event ID 4688 with process command line auditing)Sysmon (Event ID 1 — Process Create)
High Outbound network connections from Adobe Acrobat or Reader processes ►
Monitor for network connections initiated by Acrobat.exe or AcroRd32.exe to external IP addresses or domains, especially immediately after a PDF is opened. While Acrobat may legitimately connect to Adobe's update servers, connections to unknown or newly registered domains are highly suspicious and may indicate C2 callbacks from post-exploitation payloads.
EDR network telemetrySysmon (Event ID 3 — Network Connection)Firewall/proxy logs filtered by process nameDNS query logs
High Suspicious PDF files arriving via email in the past four months ►
Retroactively scan email logs and attachment repositories (December 2025 through April 2026) for PDF attachments from external senders, especially those that were opened by recipients. Resubmit suspicious PDFs to sandbox environments with the unpatched version of Acrobat to identify exploitation attempts. Pay particular attention to PDFs associated with reported anomalies or user-reported 'crashes' of Acrobat.
Email gateway logs (Proofpoint, Mimecast, Microsoft Defender for Office 365)Email attachment quarantine/archiveSandbox detonation platforms
Medium Adobe Acrobat or Reader crashes or unexpected termination events ►
Exploitation attempts — both successful and failed — often cause application crashes. Search Windows Event Logs and crash telemetry for Acrobat.exe or AcroRd32.exe crash events (Application Error, Windows Error Reporting) from December 2025 onward. Correlate crash timestamps with PDF file opens and subsequent suspicious activity.
Windows Application Event Log (Event ID 1000 — Application Error)Windows Error Reporting (WER) logsEDR process crash/fault telemetry
High Unexpected file writes or registry modifications by Acrobat processes ►
Monitor for Acrobat.exe or AcroRd32.exe writing executable files (.exe, .dll, .bat, .ps1) to disk, particularly to temp directories, user profile folders, or startup locations. Also look for registry modifications to Run/RunOnce keys or scheduled task creation originating from Adobe processes, which would indicate persistence mechanisms deployed after exploitation.
