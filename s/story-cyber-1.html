APT41 Deploys Undetectable Backdoor to Steal Cloud Credentials Across AWS, Azure, and GCP

Critical ⚠ No Fix Yet

China-backed threat group APT41 is deploying a novel backdoor with zero antivirus detections to systematically harvest cloud credentials from AWS, Google Cloud, Azure, and Alibaba Cloud environments. The campaign uses typosquatted domains mimicking legitimate cloud services for command-and-control communication, making network-based detection extremely difficult. Compromised cloud credentials grant attackers direct access to victim cloud infrastructure without needing persistent access to the original host.

📅 Apr 13, 2026 11:08 AM EDT| Source: darkreading ↗
▶Attack Flow
APT41 Zero-Detection Backdoor for Multi-Cloud Credential Harvesting
APT41 deploys a custom-packed backdoor that evades all signature-based detection engines, which then searches well-known filesystem paths and environment variables for cloud CLI credential files (e.g., ~/.aws/credentials, ~/.config/gcloud/, Azure accessTokens.json, Alibaba Cloud keys) and exfiltrates them over C2 channels disguised via typosquatted domains that mimic legitimate cloud provider infrastructure.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your developers or CI/CD systems store long-lived cloud access keys in local credential files (e.g., ~/.aws/credentials, ~/.config/gcloud/)
⚠Your environment uses environment variables to store cloud access keys (e.g., AWS_ACCESS_KEY_ID, ALIBABA_CLOUD_ACCESS_KEY_ID) on servers or workstations
⚠Your organization operates in industries historically targeted by APT41 (technology, healthcare, telecommunications, gaming, government)
⚠Your endpoints lack behavioral detection (EDR) and rely solely on signature-based antivirus
✓You exclusively use short-lived, dynamically issued credentials (e.g., IAM Roles, Workload Identity, Managed Identity) with no static keys on disk
✓Your environment has no cloud platform usage (AWS, Azure, GCP, or Alibaba Cloud) and no cloud CLI tools installed
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Audit all endpoints (developer workstations, CI/CD runners, jump servers) for the presence of static cloud credential files: search for ~/.aws/credentials, ~/.config/gcloud/credentials.db, ~/.config/gcloud/application_default_credentials.json, ~/.azure/accessTokens.json, and environment variables containing ACCESS_KEY.
  • 2Rotate all cloud access keys and tokens immediately — in AWS: `aws iam list-access-keys` then `aws iam create-access-key` / `aws iam delete-access-key`; in GCP: rotate service account keys via Cloud Console; in Azure: regenerate service principal credentials.
  • 3Review CloudTrail (AWS), Cloud Audit Logs (GCP), and Azure Activity Logs for anomalous API calls from unfamiliar IP addresses or geolocations using your service credentials within the past 90 days.
  • 4Implement DNS monitoring to flag recently registered domains that closely resemble your cloud provider endpoints (e.g., subtle misspellings of amazonaws.com, googleapis.com, azure.com, aliyuncs.com).
  • 5Deploy or verify EDR with behavioral detection capabilities on all endpoints that have cloud CLI tools installed, prioritizing developer workstations and build servers.
● Strategic This week & beyond
  • 1Eliminate long-lived static credentials across all cloud platforms: migrate to IAM Instance Roles and STS AssumeRole (AWS), Workload Identity Federation (GCP), and Managed Identity (Azure) for machine-to-machine authentication.
  • 2Implement cloud-native secrets management (AWS Secrets Manager, GCP Secret Manager, Azure Key Vault) and remove all credential files from local filesystems and environment variables.
  • 3Deploy a zero-trust architecture for cloud access with Just-in-Time (JIT) privilege elevation, requiring explicit approval for sensitive API operations and enforcing MFA for all human and programmatic access where supported.
  • 4Establish continuous credential monitoring using cloud provider tools (AWS IAM Access Analyzer, GCP Policy Analyzer, Azure AD Identity Protection) to detect credential misuse patterns and unused long-lived keys.
  • 5Implement DNS-layer security controls (DNS filtering, passive DNS monitoring) to detect and block typosquatted domains, and integrate threat intelligence feeds that track newly registered lookalike domains.
📚Concept BriefingClick a tag to expand
Cloud Credential Harvesting
The act of stealing authentication tokens and access keys that grant access to cloud platforms like AWS, Azure, and GCP, typically by reading files or environment variables where cloud CLI tools store them.
This is APT41's primary objective — systematically locating and exfiltrating cloud credential files from compromised hosts to gain direct access to victim cloud environments.
Zero-Detection Malware
Malware that has not been flagged by any antivirus or detection engine (e.g., zero detections on VirusTotal), typically achieved through custom packing, code obfuscation, or entirely novel code.
APT41's backdoor achieved zero antivirus detections at the time of discovery, meaning signature-based defenses offer no protection and behavioral or anomaly-based detection is required.
Typosquatting for C2
A technique where attackers register internet domain names that closely resemble legitimate services by introducing subtle misspellings or character substitutions, then use them for command-and-control communication.
APT41 registers domains mimicking cloud provider infrastructure to make credential exfiltration traffic appear as normal DNS queries, evading network-level security monitoring.
Static vs. Ephemeral Credentials
Static credentials are long-lived access keys stored on disk that remain valid until manually rotated; ephemeral credentials are short-lived tokens automatically issued and expired, significantly reducing the window of exploitation.
The entire attack relies on static credentials being stored in predictable file locations — transitioning to ephemeral credentials eliminates the primary target of APT41's backdoor.
Nation-State APT Attribution
Advanced Persistent Threat (APT) groups attributed to nation-state sponsors are characterized by sophisticated tooling, long-term operational patience, and strategic targeting aligned with state interests.
APT41's attribution to China and its dual espionage/financial motivation explains the broad multi-cloud targeting and investment in developing zero-detection capabilities.
🔍Threat Hunting Signals
High Unexpected reads of cloud credential files ►
Monitor for processes other than known cloud CLI tools (aws, gcloud, az) or IDEs accessing ~/.aws/credentials, ~/.config/gcloud/*, ~/.azure/accessTokens.json, or Alibaba Cloud config paths. On Linux, use auditd rules on these paths; on Windows, use Sysmon FileAccess events. Unusual reader processes (e.g., unknown binaries, shell scripts, or packed executables) indicate potential credential harvesting.
Linux auditdSysmon (Event ID 11, 15)EDR file access telemetrymacOS Endpoint Security Framework
High DNS queries to recently registered cloud-lookalike domains ►
Look for DNS resolutions of domains that closely resemble legitimate cloud provider domains (amazonaws.com, googleapis.com, azure.com, aliyuncs.com) but with character substitutions, additions, or alternative TLDs. Cross-reference against WHOIS data for domains registered in the past 90 days. Passive DNS logs and DNS firewall query logs are primary sources.
DNS server query logsDNS firewall/proxy logsPassive DNS feedsNetwork flow data
High Anomalous cloud API calls from unfamiliar source IPs ►
Review cloud audit logs for API calls (especially sts:GetCallerIdentity, iam:ListUsers, s3:ListBuckets, compute.instances.list) originating from IP addresses or geolocations not associated with your organization's known egress points or VPN ranges. This may indicate stolen credentials being used from attacker infrastructure.
AWS CloudTrailGCP Cloud Audit LogsAzure Activity LogAlibaba Cloud ActionTrail
Medium Environment variable enumeration by unknown processes ►
Detect processes querying environment variables related to cloud credentials (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, GOOGLE_APPLICATION_CREDENTIALS, AZURE_CLIENT_SECRET, ALIBABA_CLOUD_ACCESS_KEY_ID). On Linux, monitor /proc/*/environ access or use eBPF tracing; on Windows, monitor process creation events with environment variable inheritance.

Medium Packed or obfuscated binaries with outbound network connections ►
Hunt for recently dropped executables with high entropy (indicating packing/obfuscation) that establish outbound connections, particularly to non-standard ports or newly registered domains. Correlate file creation times with subsequent network connections from the same process.
