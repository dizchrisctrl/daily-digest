UNC6692 Weaponizes Microsoft Teams Helpdesk Impersonation to Drop Custom SNOW Malware

High ⚠ Mitigation Only

A newly identified threat group tracked as UNC6692 is impersonating IT helpdesk staff via Microsoft Teams to socially engineer employees into accepting chat sessions that ultimately lead to deployment of a custom malware suite called SNOW. The campaign exploits the inherent trust employees place in internal collaboration platforms, requiring no software vulnerabilities—only human compliance. Organizations that use Microsoft Teams for IT support workflows are directly targeted.

📅 Apr 23, 2026 2:16 PM EDT| Source: The Hacker News ↗
▶Attack Flow
UNC6692 – Microsoft Teams Helpdesk Impersonation Campaign Deploying SNOW Malware
UNC6692 creates or compromises accounts resembling internal IT helpdesk personnel, then initiates Microsoft Teams chat sessions with targeted employees using pretexts like "we need to fix something on your machine." Once the victim engages, the attacker convinces them to run remote access tools or download payloads, resulting in deployment of the custom SNOW malware suite on the victim's host.

❓Am I Affected?

You are likely at risk if any of the following apply:

⚠Your organization uses Microsoft Teams as a primary communication platform for IT support or helpdesk workflows
⚠Your Microsoft Teams tenant allows chat invitations from external organizations or unverified accounts
⚠Your employees have not received specific training on social engineering via collaboration platforms (as distinct from email phishing)
⚠Your environment permits users to install or run remote access tools (e.g., Quick Assist, AnyDesk, TeamViewer) without administrative approval
✓Your organization has disabled external access in Microsoft Teams and enforces strict allow-listing for federated domains
✓Your environment blocks unapproved remote access tool execution via application control policies
🛠Fix & Recommendations
⚠ Immediate 24–72 hours
  • 1Restrict Microsoft Teams external access: In the Teams Admin Center, navigate to Users > External access and set 'Allow only specific external domains' or disable external access entirely if not business-critical. Block all unmanaged external accounts from initiating chats.
  • 2Disable or restrict Quick Assist and similar remote access tools: Use Microsoft Intune or Group Policy to block execution of Quick Assist (quickassist.exe), AnyDesk, TeamViewer, and other remote support tools on endpoints where they are not explicitly required.
  • 3Issue an immediate internal alert to all employees informing them that IT helpdesk will never initiate unsolicited Teams chat sessions requesting remote access, software downloads, or credential input — and provide a verified channel for employees to confirm helpdesk requests.
  • 4Hunt for anomalous Teams activity: Review Microsoft 365 Unified Audit Logs for unusual 1:1 chat initiation patterns, especially from newly created or external accounts targeting multiple users in a short timeframe.
  • 5Scan endpoints for known SNOW malware indicators: Deploy EDR sweeps looking for unrecognized remote access tool installations, unusual scheduled tasks, and suspicious PowerShell or scripting activity following recent Teams interactions.
● Strategic This week & beyond
  • 1Extend phishing awareness training to cover collaboration platform social engineering: Update security awareness programs to explicitly address threats arriving via Teams, Slack, and similar tools — not just email.
  • 2Implement application allow-listing on endpoints: Use Microsoft Defender Application Control (WDAC) or AppLocker to prevent execution of unauthorized remote access tools and unknown binaries.
  • 3Deploy Conditional Access policies requiring MFA and compliant devices for Teams access, reducing the risk of compromised or spoofed accounts being used to impersonate internal staff.
  • 4Establish a verified IT support identity mechanism: Implement a process where all legitimate helpdesk interactions are initiated through a ticketing system and employees can verify helpdesk identity via a secondary trusted channel (e.g., corporate directory phone number).
  • 5Monitor and alert on Microsoft Teams administrative events: Integrate Teams audit logs with your SIEM and create detection rules for external account creation, guest access changes, and anomalous chat patterns targeting sensitive user groups.
📚Concept BriefingClick a tag to expand
Trusted Platform Social Engineering
A class of attack where adversaries exploit the implicit trust users place in enterprise collaboration platforms to deliver social engineering lures that bypass traditional email-focused security training.
UNC6692's entire attack chain relies on victims trusting that messages inside Microsoft Teams are legitimate, turning the collaboration tool into an intrusion vector.
Helpdesk Impersonation
A social engineering technique where attackers pose as IT support personnel to convince victims to grant remote access, run commands, or install software under the guise of troubleshooting.
UNC6692 creates accounts resembling IT helpdesk staff and uses support-themed pretexts ('we need to fix something on your machine') to gain victim compliance.
Custom Malware Suite
Purpose-built malicious software developed by a specific threat actor, as opposed to commodity or off-the-shelf tools, indicating a higher level of investment and operational sophistication.
The SNOW malware suite is previously undocumented and attributed specifically to UNC6692, signaling a dedicated and motivated threat actor rather than an opportunistic campaign.
External Access in Teams
A Microsoft Teams configuration that controls whether users from outside your organization can initiate chats, calls, or meetings with internal users.
Overly permissive external access settings allow UNC6692 to reach internal employees from external or spoofed accounts without needing to compromise the target's tenant.
Zero Trust for Human Interactions
Extending the zero-trust security model beyond network architecture to require verification of identity and intent for all human-to-human interactions within enterprise tools.
This campaign highlights the gap between network-level zero trust and the implicit trust employees grant to messages arriving inside corporate platforms.
🔍Threat Hunting Signals
High Anomalous external or newly created Teams accounts initiating 1:1 chats with multiple internal users ►
Look for accounts with display names mimicking IT helpdesk naming conventions (e.g., 'IT Support', 'Helpdesk', 'IT Admin') that are external, guest, or recently created, and that initiate chat sessions with multiple employees within a short window. This pattern indicates potential impersonation at scale.
Microsoft 365 Unified Audit LogMicrosoft Teams Admin Center activity logsAzure AD Sign-in Logs
High Remote access tool execution following Teams chat activity ►
Correlate endpoint detection logs for the launch of remote access tools (Quick Assist, AnyDesk, TeamViewer, ScreenConnect) with preceding Microsoft Teams chat sessions on the same endpoint. A short time gap between a Teams conversation and RAT execution is a strong indicator of this attack pattern.
EDR telemetry (Microsoft Defender for Endpoint, CrowdStrike, SentinelOne)Windows Event Logs (Process Creation 4688 / Sysmon Event ID 1)Microsoft Teams message audit logs
High Suspicious file downloads or payload execution linked to Teams file-sharing ►
Monitor for files downloaded via Teams chat (especially .exe, .msi, .ps1, .bat, or archive formats) from external or unrecognized senders, followed by execution on the endpoint. The SNOW malware deployment likely involves a file transfer within or adjacent to the Teams conversation.
Microsoft Defender for Endpoint file creation eventsMicrosoft Cloud App Security (Defender for Cloud Apps)Proxy/web gateway logs
Medium Unusual scheduled tasks or persistence mechanisms created after Teams-initiated remote sessions ►
After SNOW malware deployment, look for newly created scheduled tasks, registry run keys, or services on endpoints where a remote access tool was recently active. These persistence mechanisms may use innocuous-sounding names to blend in.
Sysmon Event IDs 11, 12, 13 (file/registry creation)Windows Event Log Task Scheduler (Event ID 106, 140)EDR persistence detection alerts
Medium Guest or external user additions to the Azure AD tenant without corresponding IT tickets ►
Review Azure AD audit logs for guest user invitations or external identity creations that were not initiated through a documented IT process. UNC6692 may compromise or create accounts to appear as internal staff.
